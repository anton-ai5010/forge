# FORGE v5.0.0 — Руководство пользователя

## Что такое Forge

Forge — это плагин для Claude Code, который решает три главные проблемы:

1. **Claude забывает контекст** — Forge использует трёхуровневую систему контекста (L0/L1/L2). L0 (~200 токенов) автоматически инжектируется в каждый промпт через hook — даже при сжатии контекста Claude мгновенно восстанавливается.

2. **Claude повторяет ошибки** — Forge ведёт `.forge/dead-ends/` с провальными подходами по темам. Перед любым фиксом Claude проверяет — не пробовали ли мы это уже? Записи создаются автоматически при неудачах.

3. **Claude не знает стадию проекта** — `.forge/index.yml` содержит цель, фазу, прогресс, текущую задачу, результаты последней сессии. Открываете проект через неделю — Claude сразу знает где вы остановились.

Плюс Forge даёт структурированные процессы: от идеи через дизайн и план до готового кода с тестами. Каждый скилл работает с экспертной ролью, которая настраивает Claude на конкретную задачу. Вместо чтения всего кода (~40k+ токенов) Claude читает компактную документацию (~200-800 токенов).

## Установка

Forge устанавливается как обычный плагин Claude Code:

```bash
# Клонируйте репозиторий
git clone https://github.com/anthropics/forge

# Перейдите в директорию плагина
cd forge/forge-plugin

# Установите плагин (создаст симлинк в ~/.claude/plugins/)
# Способ зависит от вашей ОС - см. документацию Claude Code
```

После установки перезапустите Claude Code. Плагин Forge будет доступен автоматически во всех проектах.

## Первый запуск

### Шаг 1: Инициализация проекта

Откройте ваш проект в Claude Code и выполните:

```
/forge:init
```

Claude:
- Просканирует структуру вашего проекта
- Обнаружит инфраструктуру: Docker (локальный и удалённый), базы данных, nginx, systemd, cron, API
- Спросит, какие файлы являются "красными зонами" (критичные, требующие особой осторожности)
- Определит язык/стек и загрузит соответствующие паттерны
- Создаст структуру документации в папке `.forge/`

Инициализация использует параллельных субагентов для сканирования — это быстро даже на больших проектах.

### Что создаётся при инициализации

```
.forge/
├── index.yml              # L0: ТОЧКА ВХОДА (~200 токенов), инжектируется в каждый промпт
├── status.md              # Что работает / сломано / заблокировано
├── decisions.md           # Ключевые технические решения и ПОЧЕМУ
├── dead-ends/             # Провальные подходы ПО ТЕМАМ (один файл на тему)
├── journal.md             # Последние 5-7 сессий с деталями
├── journal-archive/       # Старые сессии (не читается автоматически)
├── map.yml                # Карта проекта: структура, количество файлов, красные зоны
├── conventions.yml        # Правила: стиль кода, паттерны, стек-хинты
├── infrastructure.yml     # Инфраструктура: Docker, БД, nginx, cron, API
├── plans/                 # Дизайн-документы и планы реализации
└── library/               # Машинночитаемые спецификации файлов
    └── */spec.yml
```

### Пример инициализации

```
Вы: /forge:init

Claude: Сканирую проект параллельными агентами...

[Агент 1] Структура: 5 директорий, 23 файла
[Агент 2] Стек: Python 3.12, FastAPI, PostgreSQL, Redis
[Агент 3] Инфраструктура: Docker Compose (3 сервиса), nginx reverse proxy
[Агент 4] Конвенции: snake_case, pytest, alembic миграции

Какие файлы отметить как красные зоны?

Красные зоны — это файлы:
- От которых зависит production (точное поведение критично)
- Сложные и хрупкие (высокий риск сломать)
- Требующие тщательного ревью перед изменением

Например: core алгоритмы, production конфиги, модели данных

Перечислите пути к файлам (по одному в строке) или 'none':

Вы: src/core/trading_engine.py
src/strategies/production_strategy.py
none

Claude:
✓ FORGE документация инициализирована

Создано:
- .forge/index.yml (L0 — точка входа, ~200 токенов)
- .forge/map.yml (5 директорий, 2 красные зоны)
- .forge/conventions.yml (python + stack hints)
- .forge/infrastructure.yml (docker, nginx, postgresql)
- .forge/status.md, decisions.md, journal.md
- .forge/dead-ends/ (провальные подходы)
- .forge/library/ (5 директорий документированы)

Ваш проект теперь имеет контекстную документацию.
```

## Трёхуровневая система контекста (L0/L1/L2)

Forge не грузит всё подряд — он подаёт контекст порциями, по необходимости:

### L0 — Автоматический инжект (~200 токенов)

**Что:** Содержимое `.forge/index.yml` — цель проекта, стадия, текущая задача, список dead-ends тем.

**Как:** Hook автоматически подставляет в КАЖДЫЙ ваш промпт. Нулевая задержка, 100% надёжность.

**Зачем:** Даже при сжатии контекста Claude мгновенно знает где мы и что делаем. Плюс hook напоминает Claude записывать решения, тупики и уроки (doc discipline).

### L1 — По тегам (~500-2000 токенов)

**Что:** Релевантные файлы из `.forge/` — conventions.yml, конкретный dead-end, spec.yml нужной директории.

**Как:** Claude сам решает что подгрузить на основе тегов и контекста задачи.

**Зачем:** Не перегружаем промпт, но Claude имеет доступ к нужной информации.

### L2 — Полные документы (редко)

**Что:** Полные планы, архивные сессии, детальные спецификации.

**Как:** Загружается только когда Claude явно нужна детальная информация.

**Зачем:** Для глубокого анализа, ревью планов, разбора сложных проблем.

## Команды

| Команда | Что делает |
|---------|-----------|
| `/forge:start` | Полная загрузка контекста в начале сессии — цель, стадия, dead-ends, решения, журнал |
| `/forge:init` | Создаёт структуру `.forge/` для проекта — сканирует код, инфраструктуру, стек |
| `/forge:sync` | Обновляет `.forge/` по git diff после изменений в коде |
| `/forge:brainstorm` | Превращает идею в продуманный дизайн через диалог с уточнениями и выбором подхода |
| `/forge:write-plan` | Создаёт детальный план реализации с мелкими задачами по 2-5 минут каждая |
| `/forge:execute-plan` | Выполняет готовый план пакетами по 3 задачи с паузами для фидбека |
| `/forge:discover` | Ищет плагины в маркетплейсе Claude Code которые могут ускорить реализацию плана |
| `/forge:validate` | Проверяет соответствие кода плану и документации (read-only, ничего не меняет) |
| `/forge:cleanup` | Анализирует качество кода параллельными агентами и предлагает исправления |
| `/forge:unblocker` | Помогает разобраться в проекте когда застрял — сканирует, показывает проблемы, вытаскивает задачи |
| `/forge:design` | UI/UX дизайн-система — компоненты, палитры, типографика, accessibility |
| `/forge:security-review` | Аудит безопасности по OWASP чеклисту — инъекции, аутентификация, криптография |
| `/forge:api-design` | Проектирование REST API контрактов — эндпоинты, схемы, версионирование |
| `/forge:migrate` | Планирование миграций базы данных — схемы, откаты, zero-downtime |
| `/forge:deploy` | Настройка деплоя — Docker, CI/CD пайплайны, мониторинг |

## Workflow

### Старт нового проекта

**Шаг 1: Инициализация**
```
/forge:init
```

**Шаг 2: Первая фича**
```
/forge:brainstorm
```
Опишите что хотите построить. Claude задаёт вопросы на русском, одобряете дизайн, автоматически создаётся план.

**Шаг 3: Реализация**
Выбираете способ выполнения (Subagent-Driven, Batch, Autonomous). Claude пишет код с тестами.

**Шаг 4: Validate**
```
/forge:validate
```
Проверьте что всё соответствует плану и задокументировано.

**Шаг 5: Sync**
```
/forge:sync
```
Обновите документацию после завершения (если validate нашёл проблемы).

### Добавление новой фичи

**1. Brainstorm — от идеи к дизайну**
```
Вы: /forge:brainstorm
Хочу добавить кэширование результатов API

Claude: [Загружает контекст из .forge/, роль: Senior Software Architect]
Уточню: кэш для всех эндпоинтов или только для определённых?

Вы: Только для /api/indicators/*

Claude: Какое время жизни кэша?

Вы: 5 минут

Claude: [Задаёт ещё вопросы, предлагает подходы]

Вы: [Одобряете дизайн]

Claude: Дизайн сохранён. Создаю план реализации.
```

**2. Write Plan — детальный план**
```
Claude: План готов: .forge/plans/2026-02-15-api-caching.md

12 задач:
- Task 1: Установить redis-py
- Task 2: Создать конфиг для Redis
- Task 3: Написать тест для cache decorator
...

Выберите способ выполнения.

Вы: Subagent-Driven
```

**3. Execute — реализация через subagents**
```
Claude: Использую subagent-driven-development.

Task 1: Установить redis-py
[Запускает implementer subagent]
Implementer: Установка завершена.
[Запускает spec reviewer]
Spec reviewer: ✓ Соответствует спецификации
[Запускает code quality reviewer]
Code reviewer: ✓ Качество хорошее
[Запускает forge-documenter]
Documenter: Обновил .forge/library/

✓ Task 1 завершена

Task 2: Создать конфиг для Redis
[Повторяет цикл]
```

**4. Validate — проверка перед завершением**
```
Claude: Все задачи выполнены. Тесты проходят (24/24).

Запускаю валидацию перед завершением.

Вы: /forge:validate

Claude: [Проверяет код против плана и документации]

✓ Plan compliance: 12/12 tasks implemented
✓ Documentation coverage: 100%
✓ map.yml accurate

Overall Status: ALL CLEAR
```

**5. Finish — завершение**
```
Claude: Что вы хотите сделать?

1. Слить в main локально
2. Запушить и создать Pull Request
3. Оставить ветку как есть
4. Отменить эту работу

Вы: 2

Claude: [Пушит ветку, создаёт PR с описанием]

✓ PR создан: https://github.com/...
```

### Исправление бага

**1. Systematic Debugging — найти причину**
```
Вы: Тесты падают с ошибкой "Cache key not found"

Claude: [Роль: Senior Debugger] Использую systematic-debugging.

Phase 1: Root Cause Investigation

Читаю ошибку:
[показывает stack trace]

Воспроизвожу:
pytest tests/test_cache.py::test_cache_miss -v
✓ Воспроизводится стабильно

Проверяю недавние изменения:
git log -5 --oneline
[показывает коммиты]

Последний коммит изменил cache key format.

Трейсинг:
- cache.get(key) возвращает None
- key = f"api:{endpoint}:{params}"
- Но в Redis ключи: f"api_{endpoint}_{params}"

Корневая причина: несовпадение формата ключа между get и set.
```

**2. Fix с TDD**
```
Claude: Phase 4: Implementation

Шаг 1: Создаю failing test
[Пишет тест test_cache_key_format]

Шаг 2: Запускаю — падает как ожидается
FAIL: AssertionError: Key format mismatch

Шаг 3: Фикс
[Меняет разделители в set() на ":" вместо "_"]

Шаг 4: Запускаю — проходит
PASS: test_cache_key_format
PASS: все остальные тесты (25/25)

Коммит:
git commit -m "fix: use consistent cache key format"
```

### Возвращение к проекту после перерыва

**Claude читает L0 контекст и сразу в курсе:**
```
[Вы открываете проект через неделю]

Проект: trading-bot (Python)
```

**Во время работы Claude поддерживает документацию (doc discipline):**
```
[Hook напоминает каждый промпт: записывай решения, тупики, уроки]
[После каждого действия обновляется .forge/index.yml]
[Если подход не сработал — записывается в dead-ends/]
[Если принято решение — записывается в decisions.md]
[Если контекст сжался — hook при следующем промпте подставит актуальный L0]
```

**Если забыли что делали:**
```
Вы: Что у нас было в прошлый раз?

Claude: [Читает .forge/journal.md]

2026-02-08 — Кэширование API
Did: Redis, cache decorator, тесты для всех эндпоинтов
Result: 11/12 задач готово
Next: мониторинг попаданий (hit_rate, miss_rate, eviction_count)
Files: cache/redis_client.py, api/decorators.py, tests/test_cache.py
```

## Скиллы

Скиллы — это процессы и техники, которые Claude использует автоматически. Каждый скилл задаёт экспертную роль (role framing), которая настраивает Claude на задачу. Вам не нужно их вызывать вручную (кроме случаев когда явно указано).

### Планирование

#### brainstorming
Превращает идею в продуманный дизайн. Задаёт вопросы (на русском), предлагает варианты, получает одобрение, записывает документ. Использует параллельных субагентов для исследования.

**Роль:** Senior Software Architect
**Когда:** Перед любой новой фичей или большим изменением.

#### writing-plans
Создаёт детальный план реализации с мелкими задачами по 2-5 минут каждая. Точные команды, пути к файлам, ожидаемый вывод.

**Роль:** Technical Lead
**Когда:** Автоматически после brainstorming.

#### project-unblocker
Работает как "прораб" — сканирует проект, показывает что работает / что сломано / что не доделано, и вытаскивает задачи из головы через показ конкретных вещей и сбор реакций вместо открытых вопросов. Записывает ROADMAP.md и CURRENT_TASKS.md с готовыми промптами для сессий выполнения.

**Роль:** Engineering Manager
**Когда:** "Застрял", "не знаю что делать", "с чего начать", "что не доделано".

### Выполнение

#### executing-plans
Выполняет план пакетами по 3 задачи, делает паузу для фидбека между пакетами.

**Когда:** Batch execution в отдельной сессии.

#### subagent-driven-development
Выполняет план запуская свежий subagent на каждую задачу. Двухэтапное ревью после каждой: сначала соответствие спецификации, потом качество кода.

**Когда:** Subagent-Driven в текущей сессии или Autonomous в отдельной.

### Качество

#### test-driven-development (TDD)
Железное правило: тест, смотрим как падает, пишем код, смотрим как проходит. Никакого кода без теста.

**Когда:** Всегда при написании кода. Claude использует автоматически.

#### requesting-code-review
Вызывает subagent-ревьювера для проверки кода. Находит баги, проблемы качества, несоответствия спецификации.

**Когда:** После каждой задачи в subagent-driven, перед мержем, когда застряли.

#### verification-before-completion
Запрещает говорить "готово" без запуска проверки. Нет подтверждения — нет заявлений об успехе.

**Когда:** Автоматически перед любым "тесты проходят", "готово", "работает".

#### systematic-debugging
Четырёхфазный процесс: найти корневую причину, найти паттерн, сформировать гипотезу, написать тест и фикс. Никаких случайных фиксов.

**Роль:** Senior Debugger
**Когда:** При любом баге, падении теста, неожиданном поведении.

#### code-cleanup
Анализирует качество кода параллельными агентами — мёртвый код, именование, сложность, дублирование, нарушения конвенций.

**Роль:** Code Quality Specialist
**Когда:** `/forge:cleanup` или "наведи порядок", "отрефактори".

### Специализированные

#### security-review
Аудит безопасности по OWASP Top 10 — инъекции, аутентификация, криптография, конфигурация. Параллельные агенты проверяют разные категории.

**Роль:** Security Engineer
**Когда:** `/forge:security-review`, перед релизом, при работе с auth/payments.

#### api-design
Проектирование REST API контрактов — эндпоинты, схемы запросов/ответов, версионирование, пагинация, обработка ошибок.

**Роль:** API Architect
**Когда:** `/forge:api-design`, при создании нового API или рефакторинге существующего.

#### ui-ux-design
UI/UX дизайн-система — компоненты, палитры, типографика, accessibility, адаптивность.

**Роль:** UI/UX Designer
**Когда:** `/forge:design`, при создании интерфейсов.

#### database-migrations
Планирование миграций БД — схемы, индексы, откаты, zero-downtime стратегии.

**Роль:** Database Engineer
**Когда:** `/forge:migrate`, при изменении моделей данных.

#### deployment
Настройка деплоя — Dockerfile, docker-compose, CI/CD пайплайны, мониторинг, health checks. Использует параллельных субагентов.

**Роль:** DevOps Engineer
**Когда:** `/forge:deploy`, при настройке инфраструктуры.

### Контекст и память

#### Hook: автоматический инжект L0 контекста
На КАЖДЫЙ ваш запрос hook автоматически подставляет в промпт L0 контекст из `.forge/index.yml` (~200 токенов): цель, стадия, задача, список dead-ends тем, текущая ветка, последние коммиты. Плюс напоминание о doc discipline — записывать решения, тупики и уроки. Работает только в проектах с `.forge/index.yml`.

**Когда:** Автоматически, 100% надёжность, нулевая задержка.

#### session-awareness
Отвечает за ЗАПИСЬ контекста: обновляет index.yml после каждого действия, записывает провалы в dead-ends/, решения в decisions.md, состояние в status.md.

**Когда:** При завершении шагов, ошибках, провалах подходов, архитектурных решениях.

#### forge-context
Загружает L1/L2 контекст проекта из `.forge/` при старте. Выбирает релевантные файлы по тегам.

**Когда:** Автоматически при старте сессии если есть `.forge/index.yml`.

### Инфраструктура

#### using-git-worktrees
Создаёт изолированное рабочее пространство (git worktree) для работы над фичей. Не смешивается с вашей текущей веткой.

**Когда:** Автоматически при старте реализации плана.

#### finishing-a-development-branch
Завершает работу над фичей: проверяет тесты, предлагает 4 варианта (merge локально, создать PR, оставить как есть, отменить), выполняет выбор, чистит worktree.

**Когда:** Автоматически после завершения всех задач плана.

### Мета

#### writing-skills
TDD для скиллов — пишете тест (сценарий давления с subagent), смотрите как он падает, пишете скилл, смотрите как проходит, закрываете лазейки.

**Когда:** Создание или правка скиллов. Требует явного вызова.

## Структура .forge/

### .forge/index.yml — L0 контекст (читается ВСЕГДА)

```yaml
project: trading-bot
goal: Автоматическая торговля на основе технического анализа

stage:
  phase: MVP
  progress: 11/12
  blocked: null

current:
  task: Добавить кэширование API
  branch: feature/caching
  modified:
    - cache/redis_client.py
    - api/decorators.py

session:
  started: "14:30"
  goal: Мониторинг кэша
  done:
    - Настроил Redis
    - Создал cache decorator
    - Написал тесты
  now: Добавляю hit_rate метрику
  next: miss_rate, eviction_count

last_session: "2026-02-14 — Исправил формат cache key, все тесты проходят"
```

**Для чего:** Hook подставляет это в каждый промпт (~200 токенов). Claude мгновенно знает: цель проекта, стадию, текущую задачу, что уже сделано. При сжатии контекста — перечитывает и восстанавливается.

### .forge/map.yml — карта проекта

```yaml
project: trading-bot
directories:
  indicators/:
    files: 5
    red_zone_files: 1
  strategies/:
    files: 3
    red_zone_files: 2

red_zones:
  - indicators/rsi.py
  - strategies/production_strategy.py
```

**Для чего:** Claude сразу видит структуру, количество файлов, какие критичные.

### .forge/conventions.yml — правила и стек-хинты

```yaml
language: python
stack:
  - fastapi
  - postgresql
  - redis
  - pytest

naming:
  files: snake_case
  classes: PascalCase
  functions: snake_case

structure:
  indicators: "Каждый индикатор — чистая функция в своём файле"
  strategies: "Каждая стратегия наследует BaseStrategy"

patterns:
  new_indicator: "Создать файл в indicators/, добавить экспорт в __init__.py, добавить тест"
  new_endpoint: "Создать route в routes/, написать тест в tests/routes/, добавить в OpenAPI spec"

stack_hints:
  python:
    - "Type hints обязательны"
    - "pytest + fixtures вместо unittest"
    - "async/await для I/O операций"
  sql:
    - "Alembic для миграций"
    - "Индексы на foreign keys"

decisions:
  pandas_not_polars: "Существующая кодовая база использует pandas, миграция не стоит того"
```

**Для чего:** Claude пишет код в стиле проекта, следует вашим паттернам, помнит архитектурные решения. Субагенты автоматически получают стек-хинты для языков проекта (Python, TypeScript, Go, React, SQL, data-science, devops).

**Для чего:** Быстрая картина здоровья проекта.

### .forge/decisions.md — ключевые решения

```markdown
## База данных: PostgreSQL а не SQLite
Date: 2026-03-01

Decision: PostgreSQL — SQLite не тянет >50 записей/сек
Revisit if: Проект станет однопоточным
```

**Для чего:** Claude не предложит то, что уже было решено и обосновано.

### .forge/dead-ends/ — провальные подходы по темам

```
.forge/dead-ends/
├── caching.md      # Почему мемкеш не подошёл
├── auth.md         # Почему PKCE через sessionStorage не работает
└── testing.md      # Почему моки БД дают ложные результаты
```

Claude делает `ls .forge/dead-ends/`, находит релевантный файл, читает ТОЛЬКО его.

**Для чего:** Claude не повторяет ошибки. Попробовали — не работает — записали — больше не пробуем.

### .forge/journal.md — история сессий

```markdown
## 2026-02-15 — Кэширование API
Did: Redis, cache decorator, тесты для всех эндпоинтов
Result: 11/12 задач, осталось мониторинг
Next: hit_rate, miss_rate, eviction_count
Files: cache/redis_client.py, api/decorators.py, tests/test_cache.py
```

**Для чего:** Контекст между сессиями. Открыл проект через неделю — Claude читает последнюю запись и понимает где остановились.

### .forge/library/[folder]/spec.yml — что делает каждый файл

```yaml
purpose: "Индикаторы технического анализа — чистые функции"
files:
  macd.py:
    intent: "Рассчитать MACD индикатор (быстрая EMA - медленная EMA + сигнальная линия)"
    inputs:
      - "df: DataFrame"
      - "fast: int = 12"
      - "slow: int = 26"
    outputs: "DataFrame с колонками macd, signal, histogram"
    depends_on:
      - pandas
    red_zone: false
```

**Для чего:** Claude понимает назначение файла, его интерфейс, зависимости — без чтения кода.

### .forge/plans/[date]-[feature].md — дизайн и планы

Дизайн-документы из brainstorming и детальные планы реализации из writing-plans.

**Для чего:** История решений, можно вернуться к плану, понять почему сделали так.


## Как проверить что всё работает

### Шаг 1: Инициализация нового проекта

Откройте проект в новой сессии Claude Code:

```
/forge:init
```

Проверьте что создались файлы:
```bash
ls .forge/
# Должно быть: index.yml, status.md, decisions.md, dead-ends/,
# journal.md, journal-archive/, map.yml, conventions.yml,
# infrastructure.yml, plans/, library/
```

### Шаг 2: Проверка L0 контекста при старте сессии

Откройте тот же проект в НОВОЙ сессии. Claude должен:
1. Автоматически получить L0 контекст из `.forge/index.yml` через hook
2. Сказать: "Проект: X, стадия: Y, текущая задача: Z"
3. НЕ спрашивать "что делаем?" — он уже знает

### Шаг 3: Проверка live-обновления

Попросите Claude сделать что-нибудь (добавить функцию, починить баг). После выполнения проверьте:

```bash
cat .forge/index.yml
```

В секции `session` должно обновиться:
- `done:` — что было сделано
- `now:` — текущее действие
- `modified:` — какие файлы тронуты

### Шаг 4: Проверка dead-ends

Попросите Claude починить что-то сложное. Если подход не сработает, проверьте:

```bash
ls .forge/dead-ends/
cat .forge/dead-ends/<тема>.md
```

Должен появиться файл с описанием: что пробовали, почему не сработало, что делать вместо.

### Шаг 5: Проверка восстановления контекста

В длинной сессии (когда контекст начинает сжиматься) спросите:
```
Что мы сегодня делали?
```

Claude должен перечитать `.forge/index.yml` и ответить по секции `session`, а не сказать "я не помню".

### Шаг 6: Проверка между сессиями

Закройте сессию, откройте новую в том же проекте. Claude должен:
1. Получить L0 из `index.yml` — знать цель, стадию, последнюю сессию
2. Загрузить L1 из `journal.md` — знать что делали в прошлый раз
3. Предложить: "Продолжаем с [задача]. Напоминаю: [подход X] не работает."

### Что проверять при проблемах

| Проблема | Проверить |
|---|---|
| Claude не знает стадию проекта | `cat .forge/index.yml` — есть ли stage? |
| Claude повторяет ошибку | `ls .forge/dead-ends/` — записана ли она? |
| Claude не помнит что делал | `cat .forge/index.yml` — обновляется ли session? |
| Claude не знает решений | `cat .forge/decisions.md` — записаны ли они? |
| Новая сессия без контекста | `cat .forge/journal.md` — есть ли записи? |
| Claude не знает инфраструктуру | `cat .forge/infrastructure.yml` — создан ли файл? |

## Частые вопросы

### Как Claude запоминает что делал?

Три механизма:
1. **L0 Hook** — на каждый ваш запрос содержимое `.forge/index.yml` (~200 токенов) автоматически попадает в промпт. 100% надёжно — данные уже в промпте, невозможно пропустить.
2. **session-awareness** — обновляет `index.yml` после каждого действия. При сжатии контекста hook при следующем запросе снова подставит актуальный L0.
3. **Doc discipline** — hook напоминает Claude каждый промпт записывать решения, тупики и уроки. Ничего не теряется.

### Как Claude не повторяет ошибки?

Hook подставляет список тем из `.forge/dead-ends/` в каждый запрос (часть L0). При вызове скиллов (brainstorming, debugging) подгружается полное содержимое dead-ends (L1). Записи создаются скиллом `session-awareness` при провалах.

### Что такое /forge:start?

Полная загрузка контекста в начале сессии — дашборд: цель, стадия, что работает/сломано, dead-ends, журнал. Запускать один раз при открытии проекта. Дальше hook поддерживает L0 контекст автоматически.

### Что такое стек-хинты?

Субагенты автоматически получают паттерны специфичные для стека проекта из `.forge/conventions.yml`. Например:
- **Python:** type hints, pytest fixtures, async/await
- **TypeScript:** strict mode, zod validation, barrel exports
- **Go:** error wrapping, context propagation, table-driven tests
- **React:** hooks rules, memo patterns, suspense boundaries
- **SQL:** index strategy, migration safety, query optimization
- **DevOps:** multi-stage builds, health checks, graceful shutdown

Хинты определяются автоматически при `/forge:init` на основе стека проекта.

### Что такое role framing?

Каждый скилл начинает работу с установки экспертной роли. Например:
- Brainstorming: "Senior Software Architect"
- Security Review: "Security Engineer, OWASP specialist"
- Debugging: "Senior Debugger"
- Deploy: "DevOps Engineer"

Это настраивает Claude на правильный образ мышления для конкретной задачи.

### Нужно ли вручную обновлять документацию?

`session-awareness` обновляет `index.yml`, `dead-ends/`, `decisions.md` во время работы. `/forge:sync` обновляет `library/`, `map.yml`, `journal.md` — после завершения фичи.

### Что если я изменил код но забыл запустить sync?

Ничего страшного. Запустите `/forge:sync` когда вспомните — он посмотрит на git diff с момента последнего sync и обновит документацию.

### Как узнать что код соответствует плану и документации?

Используйте `/forge:validate`. Эта команда проверяет:
- Все ли задачи из плана реализованы
- Все ли файлы задокументированы в `.forge/library/`
- Актуальны ли зависимости (imports) в spec.yml
- Совпадают ли счётчики файлов в map.yml с реальностью

Validate только проверяет и отчитывается, ничего не меняет. Если нашлись проблемы — исправьте код вручную или запустите `/forge:sync` для обновления документации.

**Когда запускать:** Перед мержем, перед code review, после реализации фичи, когда что-то кажется несогласованным.

### Можно ли редактировать файлы в .forge/ вручную?

**Да.** Особенно полезно редактировать `conventions.yml` (добавить новые паттерны, стек-хинты) и `map.yml` (отметить новые красные зоны). Но обычно `/forge:sync` делает всё сам.

### Что такое "красная зона" и зачем она нужна?

Красная зона — файл, который требует особой осторожности:
- Production код, от которого зависит работающая система
- Сложный алгоритм, который легко сломать
- Файл с которым уже были проблемы

Claude видит красную зону и:
- Предупреждает вас перед изменением
- Делает более тщательное ревью
- Обязательно пишет тесты перед изменением

### Как работает Subagent-Driven Development?

Claude запускает **свежий subagent** на каждую задачу:
1. **Implementer** — пишет код, тесты, коммитит
2. **Spec reviewer** — проверяет соответствие спецификации (ничего лишнего, ничего не забыто)
3. **Code quality reviewer** — проверяет качество (читаемость, безопасность, паттерны)
4. **Forge documenter** — обновляет документацию

После каждого шага — проверка. Если нашли проблемы — фиксим и проверяем заново.

### Зачем нужны worktrees?

Git worktree — изолированная копия репозитория в отдельной папке. Работаете над фичей в worktree, ваша основная ветка остаётся нетронутой. Можете переключаться между задачами без stash/commit/reset.

### Можно ли использовать Forge на существующем проекте?

**Да.** Запустите `/forge:init` — он просканирует проект (включая инфраструктуру) и создаст документацию для текущего состояния.

### Что если проект очень большой (сотни файлов)?

Forge особенно полезен для больших проектов! L0 контекст всего ~200 токенов. Вместо чтения сотен файлов (~100k+ токенов) Claude читает компактную документацию и подгружает L1/L2 только по необходимости.

### Нужно ли коммитить .forge/ в git?

**Да, рекомендуется.** Документация полезна не только Claude, но и другим разработчикам. Infrastructure.yml и conventions.yml помогают новым участникам быстро понять проект.

### Что делать если план слишком большой?

Brainstorming автоматически предложит разбить дизайн если он больше 150 строк. Лучше несколько маленьких фич чем одна огромная.

### Можно ли изменить красную зону после инициализации?

**Да.** Отредактируйте `.forge/map.yml`:
```yaml
red_zones:
  - path/to/file.py   # добавьте/уберите файлы здесь
```

Обновите счётчик `red_zone_files` для соответствующей директории.

### Как добавить свой паттерн в conventions.yml?

Отредактируйте файл:
```yaml
patterns:
  new_api_endpoint: "Создать route в routes/, написать тест в tests/routes/, добавить в OpenAPI spec"
```

Claude будет следовать этому паттерну при создании новых эндпоинтов.

### Что если я не хочу использовать TDD?

Forge построен вокруг качественных практик включая TDD. Если вы пропускаете тесты, Claude будет напоминать о них. Вы можете игнорировать напоминания, но качество кода может пострадать.

### Можно ли использовать Forge с другими плагинами?

**Да.** Forge хорошо работает с другими плагинами Claude Code. `/forge:discover` может предлагать установку релевантных плагинов из маркетплейса.

### Как Forge помогает при code review?

`requesting-code-review` скилл запускает специализированный subagent-ревьювер:
- Проверяет соответствие требованиям
- Находит баги и уязвимости безопасности
- Оценивает качество кода
- Проверяет соблюдение conventions.yml

Ревью происходит автоматически в subagent-driven-development.
