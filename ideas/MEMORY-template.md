# Memory System — Инструкции для агентов

> Эта директория — постоянная память проекта. Всё что записано здесь переживает сессии.
> Агенты ОБЯЗАНЫ читать этот файл перед началом работы и записывать результаты после.

---

## Структура директорий

```
.claude/memory/
├── MEMORY.md              ← ты здесь (инструкции)
├── project-state.md       ← текущее состояние проекта
├── failures/              ← что НЕ сработало и почему
├── decisions/             ← почему выбрали X а не Y
├── patterns/              ← рабочие конвенции и паттерны
├── blockers/              ← что мешает прямо сейчас
├── learnings/             ← инсайты, трюки, находки
└── todo/                  ← что нужно сделать (очередь задач)
```

## project-state.md

**Кто пишет**: Foreman после каждой фазы работы, или любой агент завершивший задачу.
**Кто читает**: ВСЕ агенты в начале работы.

Формат:
```markdown
# [Название проекта] — Состояние

## Статус: [Активная разработка | Пауза | Поддержка]
## Последнее обновление: [дата, кто обновил]

## Текущий фокус
Что делается прямо сейчас. Одно предложение.

## Что сделано (последние 5 пунктов)
- [x] Описание — дата
- [x] Описание — дата

## Следующие шаги (максимум 3)
- [ ] Описание — приоритет
- [ ] Описание — приоритет

## Стек
Языки, фреймворки, базы, инструменты.

## Активные блокеры
Нет / Список со ссылками на blockers/
```

## failures/ — Неудачные решения

**Кто пишет**: Любой агент когда решение не сработало. Inspector когда находит повторяющуюся ошибку.
**Кто читает**: Все агенты перед принятием архитектурных решений.
**Зачем**: Не наступать на те же грабли. Claude склонен предлагать одни и те же решения — этот файл говорит ему "это уже пробовали, не работает".

Формат файла: `failures/YYYY-MM-DD-короткое-описание.md`

```markdown
# Неудачное решение: [название]

## Дата: YYYY-MM-DD
## Контекст
Что пытались сделать и зачем.

## Что попробовали
Конкретное решение которое реализовали.

## Почему не сработало
Конкретная причина провала. Ошибки, метрики, поведение.

## Чем заменили
Что в итоге сработало (если уже нашли замену).

## Урок
Одно предложение — что запомнить на будущее.
```

Примеры для твоих проектов:
- `2026-03-15-polymarket-arbitrage-fees.md` — Арбитраж Polymarket убит комиссией 30 марта
- `2026-02-20-yarick-accuracy-inflated.md` — 88% accuracy оказалась selection bias
- `2026-01-10-async-telegram-bot-memory-leak.md` — Утечка памяти из-за незакрытых сессий

## decisions/ — Архитектурные решения (ADR)

**Кто пишет**: Ты вручную или Foreman после утверждения плана.
**Кто читает**: Все агенты перед предложением альтернатив.
**Зачем**: Через месяц ты забудешь ПОЧЕМУ выбрал FastAPI а не Django. Агенты не будут предлагать менять стек если увидят обоснованное решение.

Формат файла: `decisions/NNNN-название.md`

```markdown
# ADR-0001: [Название решения]

## Дата: YYYY-MM-DD
## Статус: [Принято | Отменено | Заменено ADR-XXXX]

## Контекст
Какая проблема стояла. Какие ограничения были.

## Рассмотренные варианты
1. **Вариант A** — плюсы / минусы
2. **Вариант B** — плюсы / минусы
3. **Вариант C** — плюсы / минусы

## Решение
Выбрали вариант X потому что [причины].

## Последствия
Что это означает для проекта. Что нельзя делать из-за этого решения.
```

Примеры:
- `0001-fastapi-not-django.md` — Для CAME BOT: FastAPI быстрее для API-сервиса, Django overkill
- `0002-xgboost-not-neural-net.md` — Для YARICK: XGBoost на малых данных бьёт нейросети
- `0003-vless-not-vmess.md` — Для VPN: VLESS+xHTTP обходит DPI лучше чем VMess

## patterns/ — Рабочие паттерны и конвенции

**Кто пишет**: Inspector после обнаружения повторяющегося паттерна. Ты вручную для новых правил.
**Кто читает**: python-pro, backend-developer, frontend-developer — перед написанием кода.
**Зачем**: Единообразие кода. Агент не будет писать sync-код если видит что весь проект на async.

Формат: один файл `patterns/conventions.md` (пока проект маленький), потом разбивать по темам.

```markdown
# Конвенции проекта

## Код
- Type hints обязательны на всех функциях
- Async/await для любого I/O (БД, HTTP, файлы)
- Docstrings: Google-style, на русском для бизнес-логики, на английском для утилит
- Именование: snake_case для Python, camelCase для JS/TS
- Максимум 200 строк на файл, дальше — декомпозиция

## Структура проекта
- src/ — исходный код
- tests/ — тесты (зеркалят структуру src/)
- docs/ — документация
- scripts/ — утилиты для деплоя/миграций

## Обработка ошибок
- Никогда bare except
- Кастомные исключения для бизнес-логики (KeyNotFoundError, OrderExpiredError)
- Retry с exponential backoff для внешних API
- Логирование через structlog, не print

## Git
- Conventional commits: feat/fix/refactor/docs/test
- Ветки: feature/xxx, fix/xxx, от main
- Squash merge в main

## Тесты
- pytest + pytest-asyncio
- Минимум: happy path + 2 edge case на функцию
- Моки для внешних API, реальная БД для интеграционных
```

## blockers/ — Активные блокеры

**Кто пишет**: Любой агент при обнаружении проблемы которая мешает продвижению. Ты вручную.
**Кто читает**: Foreman перед планированием задач.
**Зачем**: Foreman не назначит задачу которая зависит от заблокированной. Ты видишь что мешает.

Формат файла: `blockers/YYYY-MM-DD-описание.md`

```markdown
# Блокер: [описание]

## Дата обнаружения: YYYY-MM-DD
## Блокирует: [что не может продвинуться]
## Серьёзность: [Критический | Высокий | Средний]

## Описание
Что именно мешает.

## Возможные решения
1. ...
2. ...

## Статус: [Открыт | В работе | Решён]
## Решён: YYYY-MM-DD (если решён — переместить в learnings/)
```

## learnings/ — Инсайты и находки

**Кто пишет**: Любой агент после завершения нетривиальной задачи. Inspector после обнаружения паттерна.
**Кто читает**: Все агенты как справочник.
**Зачем**: Накопление знаний. "В прошлый раз Яндекс Маркет API возвращал 429 после 100 запросов/мин — нужен rate limiter".

Формат файла: `learnings/YYYY-MM-DD-тема.md`

```markdown
# Инсайт: [тема]

## Дата: YYYY-MM-DD
## Категория: [API | Производительность | Данные | Инфраструктура | Бизнес]

## Что узнали
Конкретный факт или наблюдение.

## Как применять
Когда это знание релевантно и что делать.
```

Примеры:
- `2026-03-20-yandex-market-rate-limit.md` — Лимит 100 req/min, нужен rate limiter
- `2026-03-15-xgboost-closing-odds-feature.md` — Closing odds дают +3% к accuracy
- `2026-02-28-marzban-xhttp-config.md` — Рабочая конфигурация VLESS+xHTTP для обхода ТСПУ

## todo/ — Очередь задач

**Кто пишет**: Ты, Foreman после декомпозиции.
**Кто читает**: Foreman при /build.
**Зачем**: Один источник правды о том что нужно делать. Не в голове, не в заметках — здесь.

Формат: один файл `todo/backlog.md`

```markdown
# Backlog

## Высокий приоритет
- [ ] [CAME BOT] Модуль выдачи ключей — фаза 2/4
- [ ] [YARICK] Walk-forward validation на unseen данных

- [ ] [CAME BOT] Telegram-уведомления о низких остатках

## Низкий приоритет / Идеи
```

## Инструкции для агентов

### При старте сессии:
1. Прочитай `project-state.md`
2. Просмотри `blockers/` — есть ли активные блокеры
3. Просмотри `failures/` — есть ли релевантные к текущей задаче
4. Используй `patterns/conventions.md` как стандарт кодирования

### После завершения задачи:
1. Обнови `project-state.md` — что изменилось
2. Если решение не сработало → запиши в `failures/`
3. Если обнаружил новый паттерн → запиши в `patterns/`
4. Если узнал что-то полезное → запиши в `learnings/`
5. Если задача из `todo/backlog.md` → отметь как выполненную

### При обнаружении проблемы:
1. Запиши в `blockers/`
2. Обнови `project-state.md` → секция "Активные блокеры"
3. Предложи решение

### НИКОГДА:
- Не удаляй записи из `failures/` — это история
- Не меняй `decisions/` без подтверждения пользователя
- Не игнорируй `patterns/conventions.md` — следуй конвенциям
