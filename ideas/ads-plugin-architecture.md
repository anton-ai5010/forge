# Архитектура плагина: Anton's Development System (ADS)

## Стратегический контекст

Прежде чем рисовать структуру папок — зафиксируем проблемы которые система должна решать. Не "какие агенты поставить", а "какие поведенческие паттерны изменить".

### Корневые проблемы

1. **Потеря состояния** — между сессиями Claude теряет контекст проекта. Ты вынужден каждый раз заново объяснять что за проект, где остановились, какие решения приняты.

2. **Паралич декомпозиции** — ты видишь конечную цель ("сделать модуль оплаты"), но не можешь разбить на исполняемые шаги. Не потому что не умеешь — потому что переключение между архитектурным мышлением и тактическим исполнением стоит когнитивных ресурсов.

3. **Размазанный фокус** — 4 проекта параллельно (CAME BOT, YARICK, VPN, Upwork), каждый в разной стадии. Контекст-переключение убивает продуктивность.

4. **Слепые зоны качества** — код работает, но никто не ревьюит, не тестирует, не проверяет безопасность. Баги обнаруживаются в проде.

### Принцип проектирования

Плагин не должен быть коллекцией инструментов. Он должен быть **операционной системой для соло-разработчика** — с ролевой моделью, автоматическими проверками, и памятью которая переживает сессии.

### Ключевая идея: три роли, а не набор агентов

Вместо 30 разрозненных агентов — **три организационные роли**:

**Architect** (ты) — принимаешь решения, задаёшь направление, приоритизируешь. Это не агент, это ты.

**Foreman** (прораб) — кастомный агент-оркестратор. Получает от тебя задачу на уровне "сделай модуль оплаты", самостоятельно:
- Декомпозирует на подзадачи
- Выбирает подходящего специалиста для каждой
- Маршрутизирует выполнение
- Собирает результаты
- Обновляет project-state

**Inspector** (контролёр) — кастомный агент который автоматически запускается после каждого изменения кода:
- Code review
- Проверка безопасности
- Соответствие конвенциям проекта
- Запуск тестов

Специалисты (python-pro, data-scientist, etc.) — вызываются Foreman-ом, а не тобой напрямую. Ты общаешься только с Foreman.

## Детальное проектирование ключевых компонентов

### 1. Foreman — агент-прораб

Это центральный элемент системы. Не просто "оркестратор", а **проектный менеджер который говорит на языке архитектора**.

```yaml
# agents/foreman.md
---
name: foreman
description: |
  Прораб проекта. Вызывается когда нужно реализовать задачу любой сложности.
  Принимает задачу на архитектурном уровне, самостоятельно декомпозирует,
  выбирает специалистов, маршрутизирует выполнение, контролирует результат.
  Use when: пользователь описывает задачу но не указывает КАК её делать.
  "сделай модуль оплаты", "добавь авторизацию", "настрой деплой".
tools: Read, Write, Edit, Bash, Glob, Grep, Task
model: sonnet
memory: project
---
```

**Поведенческая модель Foreman-а:**

```
ВХОД: "Сделай модуль выдачи ключей для CAME BOT"
                    │
                    ▼
     ┌──────────────────────────┐
     │ 1. КОНТЕКСТ              │
     │ Прочитай project-state   │
     │ Пойми текущее состояние  │
     │ проекта, стек, конвенции │
     └──────────┬───────────────┘
                │
                ▼
     ┌──────────────────────────┐
     │ 2. ДЕКОМПОЗИЦИЯ          │
     │ Разбей на подзадачи:     │
     │ - Спроектировать API     │
     │ - Написать бизнес-логику │
     │ - Написать тесты         │
     │ - Подключить к маркетпл. │
     └──────────┬───────────────┘
                │
                ▼
     ┌──────────────────────────┐
     │ 3. МАРШРУТИЗАЦИЯ         │
     │ API → @api-designer      │
     │ Логика → @python-pro     │
     │ Тесты → @test-engineer   │
     │ Интегр. → @backend-dev   │
     └──────────┬───────────────┘
                │
                ▼
     ┌──────────────────────────┐
     │ 4. ИСПОЛНЕНИЕ            │
     │ Запуск через Task tool   │
     │ Каждый специалист —      │
     │ отдельный субагент       │
     └──────────┬───────────────┘
                │
                ▼
     ┌──────────────────────────┐
     │ 5. ВЕРИФИКАЦИЯ           │
     │ Вызвать @inspector       │
     │ Обновить project-state   │
     │ Доложить архитектору     │
     └──────────────────────────┘
```

**Критически важное поведение Foreman-а:**
- Всегда НАЧИНАЕТ с чтения project-state (не спрашивает "а что за проект?")
- Показывает план декомпозиции ПЕРЕД выполнением (ты можешь скорректировать)
- Каждая подзадача = отдельный Task (не в основном контексте)
- После выполнения — обновляет project-state.md

### 2. Inspector — агент-контролёр

```yaml
# agents/inspector.md
---
name: inspector
description: |
  Контролёр качества. Вызывается автоматически после изменений кода
  или по запросу через /review. Проверяет: баги, безопасность,
  производительность, конвенции проекта, тесты.
  НЕ пишет код. Только анализирует и выдаёт вердикт.
tools: Read, Grep, Glob, Bash
model: sonnet
memory: project
---
```

**Чеклист Inspector-а:**
1. Безопасность — secrets в коде? SQL injection? XSS? Открытые порты?
2. Баги — необработанные исключения? Race conditions? Off-by-one?
3. Конвенции — type hints? Docstrings? Naming? Структура файлов?
4. Тесты — есть ли для нового кода? Покрыты ли edge cases?
5. Производительность — N+1 запросы? Утечки памяти? Тяжёлые циклы?

**Agent Memory** позволяет Inspector-у накапливать знания о проекте:
- Какие паттерны используются
- Какие ошибки уже находил раньше
- Специфические конвенции проекта

### 3. Project Context — скилл управления состоянием

Это решение проблемы #1 (потеря состояния). Скилл создаёт и поддерживает набор файлов в каждом проекте:

```
.claude/
├── memory/
│   ├── project-state.md       # Текущее состояние: что сделано, что в процессе, что дальше
│   ├── decisions.md           # Архитектурные решения и почему они приняты
│   ├── patterns.md            # Паттерны и конвенции проекта (накапливает Inspector)
│   └── foreman-log.md         # Лог задач Foreman-а: что делал, результат
```

**project-state.md — формат:**
```markdown
# CAME BOT — Состояние проекта

## Статус: Активная разработка

## Текущий фокус
Модуль выдачи ключей — в процессе (фаза 2/4: бизнес-логика)

## Что сделано
- [x] Структура БД (таблицы: keys, orders, suppliers)
- [x] API endpoint: GET /orders (интеграция с Яндекс Маркет)
- [ ] Бизнес-логика выдачи ключей (IN PROGRESS)
- [ ] Тесты
- [ ] Деплой

## Стек
Python 3.12, FastAPI, PostgreSQL, SQLAlchemy, pytest

## Конвенции
- Type hints обязательны
- Async/await для I/O
- Alembic для миграций

## Блокеры
Нет активных блокеров

## Последнее обновление
2026-04-05 — Foreman завершил фазу 1 (API design)
```

**Хук session-start-context** автоматически загружает этот файл при старте сессии. Claude сразу знает где ты.

### 4. Команды — интерфейс взаимодействия

Каждая команда — чёткий entry point, не перегруженный опциями:

#### /plan [описание задачи]
Декомпозирует задачу через Foreman, показывает план, ждёт подтверждения.
```
/plan Добавить систему уведомлений о низких остатках ключей
→ Foreman читает project-state
→ Показывает план: 4 подзадачи, оценка сложности
→ "Подтвердить? Изменить? Отменить?"
```

#### /build
Запускает исполнение утверждённого плана. Foreman маршрутизирует подзадачи специалистам.
```
/build
→ Foreman запускает фазу 1 через @python-pro
→ По завершении — @inspector проверяет
→ Если ок — переходит к фазе 2
→ Обновляет project-state
```

#### /review
Ручной запуск Inspector-а на текущих изменениях.
```
/review
→ Inspector анализирует diff
→ Выдаёт отчёт: 2 бага, 1 security issue, 3 style
```

#### /visualize
Генерирует нод-схему проекта (project-visualizer скилл).
```
/visualize
→ Читает project-state + CLAUDE.md
→ Генерирует React Flow HTML
→ Открывается интерактивная карта
```

#### /research [тема]
Делегирует ресёрч подходящему researcher-агенту.
```
/research Сравни комиссии FunPay vs GGSEL для цифровых товаров
→ Маршрутизация на @competitive-analyst
→ Результат: структурированный отчёт
```

#### /status
Показывает текущее состояние проекта из project-state.md.
```
/status
→ "CAME BOT: фаза 2/4, бизнес-логика выдачи ключей"
→ "YARICK: заморожен — последняя активность 2 недели назад"
→ "VPN: конфигурация серверов завершена, следующий шаг — тестирование"
```

#### /switch [проект]
Переключает контекст на другой проект. Сохраняет текущее состояние, загружает новое.
```
/switch yarick
→ Сохраняет project-state CAME BOT
→ Загружает project-state YARICK
→ "YARICK: последняя сессия — анализ покрытия лиг. Продолжить?"
```

## Принципы проектирования

### 1. Минимум прямых обращений к специалистам
Ты общаешься с Foreman. Foreman общается со специалистами. Это сокращает когнитивную нагрузку — тебе не нужно помнить кто что умеет.

**Исключение**: Ресёрч-агенты вызываются напрямую через /research — потому что ресёрч это не исполнение, а принятие решений, и ты должен быть в процессе.

### 2. Всё что можно автоматизировать — автоматизировать
- Code review → хук после каждого Edit/Write
- Загрузка контекста → хук при старте сессии
- Обновление состояния → Foreman делает после каждой фазы
- Тесты → Inspector запускает как часть проверки

### 3. Память переживает сессии
project-state.md, decisions.md, patterns.md — коммитятся в git. Каждая сессия начинается с загрузки, заканчивается обновлением. Состояние проекта НИКОГДА не теряется.

### 4. Модульность — всё отключаемо
Не нужен Inspector? Удали файл. Не нужен data-scientist? Удали. Система работает пока есть Foreman + хотя бы 1 специалист.

### 5. Экономия токенов по умолчанию
- Специалисты на `model: sonnet` (дешевле)
- Foreman на `model: sonnet` (достаточно для маршрутизации)
- Inspector на `model: sonnet` (ревью не требует Opus)
- `model: opus` — только если ты явно попросишь для сложной архитектурной задачи

## Чего НЕ делать

1. **Не ставить все 40 агентов сразу**. Начать с Foreman + Inspector + 3-4 специалиста. Добавлять по мере необходимости.

2. **Не дублировать функциональность**. Один агент = одна роль. Не нужны и `fullstack-developer` и `backend-developer` и `python-pro` одновременно — Foreman выберет нужного.

3. **Не автоматизировать решения**. Автоматизировать исполнение — да. Решения остаются за тобой. Foreman ПОКАЗЫВАЕТ план и ЖДЁТ подтверждения.

4. **Не игнорировать Inspector**. Соблазнительно отключить автоматический ревью когда "надо быстрее". Именно в эти моменты появляются баги.

5. **Не делать plugin.json слишком амбициозным**. Лучше 5 работающих команд чем 20 полусырых.
