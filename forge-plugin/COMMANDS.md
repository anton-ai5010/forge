# FORGE Plugin Commands Reference v5.6.0

Полный справочник всех 16 команд forge-plugin с описанием назначения и использования.

## 0. /forge:start

**Назначение:** Полная загрузка контекста проекта в начале сессии

**Когда использовать:**
- Один раз при открытии проекта в новой сессии
- Когда нужен полный дашборд текущего состояния

**Что делает:**
- Через backtick-препроцессинг мгновенно загружает: index.yml, status.yml, decisions.yml, dead-ends, journal.yml, git log
- Показывает дашборд: цель, стадия, что работает/сломано, провальные подходы
- Предлагает продолжить незаконченную задачу

**Результат:** Claude полностью в контексте, готов работать без дополнительных вопросов

**Примечание:** Между запросами контекст поддерживается автоматически через UserPromptSubmit hook (инжект index.yml + dead-ends). `/forge:start` нужен только в начале.

## 1. /forge:init

**Назначение:** Инициализация структуры FORGE документации в проекте

**Когда использовать:**
- Один раз в начале разработки проекта
- При миграции старого проекта на FORGE
- Для создания контекстной документации

**Что делает:**
- Запускает 5 параллельных субагентов для сканирования проекта:
  - Структура кода и критичные файлы (red zones)
  - Конвенции проекта (язык, именование, стиль)
  - Инфраструктура (Docker, базы данных, серверы, CI/CD)
  - Продуктовый и технический контекст
  - Графовая карта кода (если graphify установлен)
- Создаёт `.forge/` директорию с:
  - `map.yml` — структура и критичные файлы (red zones)
  - `conventions.yml` — соглашения проекта (язык, именование)
  - `index.yml` — точка входа: цель, стадия, сессия, ссылки
  - `infrastructure.yml` — Docker, БД, серверы, CI/CD конфигурация
  - `library/` — зеркало документации под структуру проекта
  - `product.yml` — контекст продукта
  - `tech.yml` — технический контекст
- Генерирует всё в YAML формате

**Результат:** Полная структура контекстной документации (~2k токенов вместо ~40k+ исходного кода)

**Пример:**
```
/forge:init

Инициализировать футбольный предиктор.
Язык: Python
Red zones: indicators/smart_money.py, strategies/production.py
```

## 2. /forge:sync

**Назначение:** Обновление документации после внесения изменений в код

**Когда использовать:**
- После каждой реализованной задачи/фичи
- Перед коммитом
- Перед PR в основную ветку
- Для поддержания актуальности документации

**Что делает:**
- Определяет изменения со времени последней синхронизации (git diff)
- Обновляет `.forge/library/` для новых/изменённых файлов
- Обновляет счёты файлов в `.forge/map.yml`
- Обновляет `.forge/index.yml` со стадией и последней сессией
- Проверяет и обновляет `.forge/infrastructure.yml` (Docker, БД, серверы)
- Добавляет запись в `.forge/journal.yml`

**Результат:** Актуальная документация, готовая для следующей итерации

**Пример:**
```
/forge:sync

Я завершил Phase 9 Web Interface.
Добавлены: api/, scripts/run_server.py, тесты
Обновлены: PredictionPipeline, test suites
```

## 3. /forge:brainstorm

**Назначение:** Исследование требований и дизайна ДО начала разработки

**Когда использовать:**
- Перед любой творческой работой (новые фичи, компоненты, функциональность)
- Когда нужно спроектировать архитектуру
- Для изучения требований и вариантов решения

**Что делает:**
- Проводит интерпретивный анализ задачи
- Изучает существующий проект
- Предлагает несколько подходов к решению
- Помогает выбрать лучший путь реализации

**Результат:** Одобренный дизайн, готовый для планирования

**Пример:**
```
/forge:brainstorm

Реализовать тёмный режим в приложении.
Требования: переключение в меню, сохранение предпочтения, поддержка всех страниц.
```

## 4. /forge:write-plan

**Назначение:** Создание детального плана реализации с кусочками работы (bite-sized tasks)

**Когда использовать:**
- После одобрения дизайна (следует за brainstorm)
- Перед началом разработки
- Когда нужна пошаговая схема выполнения

**Что делает:**
- Разбивает задачу на 3-15 конкретных шагов
- Каждый шаг содержит: файлы, правила, код, тесты
- Структурирует работу для выполнения за раз
- Создаёт файл плана в `.forge/plans/`

**Результат:** Файл `.forge/plans/{дата}-{название}-plan.yml` с полным кодом

**Пример:**
```
/forge:write-plan

На основе дизайна тёмного режима (из brainstorm)
создать план реализации с 6 задачами:
1. CSS переменные
2. Context Provider
3. Theme switcher компонент
4. Тесты
5. Документация
```

## 5. /forge:execute-plan

**Назначение:** Реализация плана с контрольными точками и проверками

**Когда использовать:**
- После создания плана
- Когда нужна пошаговая реализация с верификацией

**Что делает:**
- Берёт существующий план
- Реализует задачи одну за другой
- После каждой задачи проверяет результат
- Запрашивает одобрение перед переходом к следующей
- Автоматически коммитит завершённые блоки

**Результат:** Реализованный план, все коммиты, готовый код

**Пример:**
```
/forge:execute-plan

Реализовать план из .forge/plans/2026-02-15-dark-mode-plan.yml
```

## 6. /forge:discover

**Назначение:** Поиск готовых плагинов в маркетплейсе (для ускорения разработки)

**Когда использовать:**
- После написания плана, перед выбором подхода к реализации
- Когда нужны готовые инструменты для ускорения
- Для проверки, существует ли уже то, что нужно

**Что делает:**
- Анализирует план
- Извлекает технологии, домены, паттерны
- Ищет релевантные плагины в маркетплейсе
- Рекомендует высоко рейтинговые решения
- Предлагает установить или создать кастомный плагин

**Результат:** Список рекомендованных плагинов или перечень того, что нужно создать

**Примечание:** Маркетплейс API ещё в разработке. Сейчас команда показывает, что БЫ было при наличии API.

## 7. /forge:validate

**Назначение:** Проверка соответствия кода плану и документации (read-only)

**Когда использовать:**
- Перед мёржем фичи в основную ветку
- После завершения реализации плана
- Когда что-то кажется несоответствующим
- Для верификации перед PR

**Что делает:**
- Сравнивает код с планом (все ли задачи завершены?)
- Проверяет документацию против кода (все ли файлы задокументированы?)
- Проверяет точность `.forge/map.yml` (счёты файлов совпадают?)
- Выявляет несоответствия и предлагает действия

**Результат:** Подробный отчёт о несоответствиях (если есть) с рекомендациями по исправлению

**Примечание:** Только чтение и анализ, не модифицирует файлы

**Пример:**
```
/forge:validate

Проверить что Phase 9 полностью реализована по плану
```

## 8. /forge:cleanup

**Назначение:** Анализ качества кода по всему проекту с параллельными агентами

**Когда использовать:**
- Когда код стал запутанным после быстрого прототипирования
- Перед релизом — навести порядок
- Для погашения технического долга
- Периодически для поддержания качества

**Что делает:**
- Разбивает проект на независимые области (по директориям/модулям)
- Запускает параллельных агентов для анализа каждой области
- Проверяет: мёртвый код, именование, сложность, дублирование, структуру, конвенции
- Собирает сводный отчёт по severity (high/medium/low)
- Спрашивает что исправить (всё / только critical / выборочно)
- Исправляет только после одобрения

**Результат:** Структурированный отчёт + исправления по запросу

**Примечание:** Только анализ до одобрения, red zone файлы никогда не правятся автоматически

**Пример:**
```
/forge:cleanup

Проверить качество кода во всём проекте.
Особое внимание: именование, дублирование, мёртвый код.
```

## 9. /forge:unblocker

**Назначение:** Помочь разобраться в проекте и понять что делать дальше когда застрял

**Когда использовать:**
- "Не знаю что делать дальше", "застрял", "с чего начать"
- "Что тут вообще происходит", "не понимаю проект"
- "Что не доделано", "помоги разобраться", "нужен план"
- Жалобы на проект: "не нравится", "тормозит", "криво работает"
- Любые признаки потерянности в проекте

**Что делает:**
- Фиксирует цель проекта (предлагает сам, не спрашивает)
- Глубоко сканирует проект: код, инфраструктура, документация, состояние
- Показывает что работает, что начато, что сломано
- Извлекает задачи через реакции (не через открытые вопросы)
- Формирует план с приоритетами
- Записывает ROADMAP.md и CURRENT_TASKS.md

**Результат:** ROADMAP.md + CURRENT_TASKS.md с готовыми задачами-промптами для отдельных сессий

**Важно:** Это сессия ПЛАНИРОВАНИЯ — код не пишется, баги не чинятся. Задачи формулируются как самодостаточные промпты для сессий выполнения.

**Пример:**
```
/forge:unblocker

Застрял с этим проектом, не знаю за что браться.
```

## 10. /forge:design

**Назначение:** Генерация UI/UX дизайн-системы для проекта

**Когда использовать:**
- При создании нового фронтенд-проекта или редизайне
- Когда нужна согласованная визуальная система
- Для стандартизации стилей, цветов, шрифтов, компонентов

**Что делает:**
- Использует BM25 поиск по CSV базам данных дизайн-паттернов
- Генерирует палитру цветов (primary, secondary, accent, neutral, semantic)
- Подбирает типографику (шрифты, размеры, line-height, иерархия)
- Создаёт UX гайдлайны (spacing, border-radius, shadows, анимации)
- Определяет компонентную библиотеку (кнопки, инпуты, карточки, модалки)
- Учитывает accessibility (контраст, focus states, screen readers)

**Результат:** Файл `.forge/plans/design-system-{название}.yml` с полной дизайн-системой

**Пример:**
```
/forge:design

Создать дизайн-систему для SaaS-приложения аналитики.
Стиль: минималистичный, профессиональный.
Целевая аудитория: бизнес-пользователи.
```

## 11. /forge:security-review

**Назначение:** Проверка безопасности проекта по чеклисту OWASP

**Когда использовать:**
- Перед деплоем в продакшн
- После добавления аутентификации, API, обработки данных
- Периодически для аудита безопасности
- Перед мёржем крупных фич

**Что делает:**
- Сканирует проект по 6 категориям:
  - **Secrets** — захардкоженные ключи, токены, пароли в коде и конфигах
  - **Input Validation** — SQL injection, XSS, path traversal, command injection
  - **Auth** — аутентификация, авторизация, управление сессиями
  - **API Security** — rate limiting, CORS, валидация запросов, error leaking
  - **Dependencies** — уязвимые зависимости, устаревшие пакеты
  - **Data Privacy** — PII в логах, шифрование, GDPR compliance
- Для каждой категории выдаёт PASS / FAIL с деталями

**Результат:** Структурированный отчёт PASS/FAIL по каждой категории с конкретными находками и рекомендациями

**Пример:**
```
/forge:security-review

Полный аудит безопасности перед релизом v2.0.
Особое внимание: API endpoints, обработка пользовательских данных.
```

## 12. /forge:api-design

**Назначение:** Проектирование REST API контракта

**Когда использовать:**
- При создании нового API или микросервиса
- При расширении существующего API
- Когда нужен согласованный API контракт до начала разработки

**Что делает:**
- Определяет ресурсы и их связи (entities, relationships)
- Проектирует эндпоинты (CRUD, custom actions, nested resources)
- Настраивает пагинацию (cursor-based / offset, page size)
- Стандартизирует обработку ошибок (error codes, error body format)
- Определяет стратегию версионирования (URL / header / query param)
- Описывает модели запросов и ответов (request/response schemas)
- Сохраняет контракт в `.forge/plans/api-{название}.yml`

**Результат:** Файл `.forge/plans/api-{название}.yml` с полным API контрактом

**Пример:**
```
/forge:api-design

Спроектировать API для маркетплейса цифровых товаров.
Ресурсы: товары, категории, заказы, пользователи, отзывы.
Аутентификация: JWT.
```

## 13. /forge:migrate

**Назначение:** Планирование миграций базы данных с zero-downtime стратегией

**Когда использовать:**
- При изменении схемы БД (новые таблицы, колонки, индексы)
- При breaking changes в структуре данных
- Когда нужна миграция без простоя

**Что делает:**
- Анализирует текущую схему БД и требуемые изменения
- Для breaking changes применяет expand-contract паттерн:
  1. **Expand** — добавить новое, не удаляя старое
  2. **Migrate data** — перенести данные в новую структуру
  3. **Contract** — удалить старое после полного перехода
- Генерирует forward и rollback скрипты (rollback обязателен)
- Проверяет совместимость с текущим кодом
- Оценивает время и риски миграции

**Результат:** Файл `.forge/plans/migration-{название}.yml` с планом миграции, SQL скриптами и rollback

**Пример:**
```
/forge:migrate

Разделить таблицу users на users + user_profiles.
Текущая БД: PostgreSQL, ~2M записей.
Требование: zero-downtime, обратная совместимость на время перехода.
```

## 14. /forge:deploy

**Назначение:** Настройка деплоймента: контейнеризация, CI/CD, мониторинг

**Когда использовать:**
- При первичной настройке деплоя проекта
- При переходе на контейнеризацию
- Когда нужен CI/CD пайплайн
- Для настройки health checks и rollback стратегии

**Что делает:**
- Генерирует Docker multi-stage build (оптимизация размера и безопасности)
- Настраивает CI/CD пайплайн (GitHub Actions / GitLab CI / другое)
- Определяет health check endpoints и readiness probes
- Настраивает стратегию rollback (автоматический при failed health check)
- Создаёт docker-compose для локальной разработки и staging
- Конфигурирует переменные окружения и secrets management

**Результат:** Готовые конфигурационные файлы + `.forge/plans/deploy-{название}.yml` с описанием стратегии

**Пример:**
```
/forge:deploy

Настроить деплой для Node.js API на Docker + GitHub Actions.
Staging: docker-compose на сервере.
Production: Kubernetes кластер.
Health check: /api/health
```

## 15. /forge:graph

**Назначение:** Построить или обновить графовую карту кода (зависимости, импорты, вызовы)

**Когда использовать:**
- После `/forge:init` для большого проекта (50+ файлов)
- Когда нужно понять архитектуру и связи между модулями
- Перед brainstorm/debugging для быстрой навигации по коду

**Что делает:**
- Запускает graphify для построения графа кода через tree-sitter (без LLM, быстро)
- Сохраняет граф в `.forge/graph.json`
- Генерирует отчёт `.forge/graph-report.md`
- Все скиллы Forge автоматически используют граф для навигации

**Режимы:**
- `/forge:graph` — полная сборка графа с нуля
- `/forge:graph --update` — инкрементальное обновление (только изменённые файлы, без LLM)
- `/forge:graph --query "вопрос"` — поиск по графу
- `/forge:graph --path "A" "B"` — путь между компонентами
- `/forge:graph --explain "Node"` — объяснение узла и его соседей

**Требования:** graphify (`pipx install graphifyy`)

**Автоматизация:** строится при `/forge:init`, обновляется при `/forge:sync`

**Пример:**
```
/forge:graph
/forge:graph --query "как работает авторизация?"
/forge:graph --path "UserModel" "PaymentService"
```

## Workflows

### Полный цикл разработки

```
1. /forge:init                  — инициализация проекта
   ↓
2. /forge:brainstorm            — исследование и дизайн
   ↓
3. /forge:write-plan            — детальный план
   ↓
4. /forge:execute-plan          — реализация
   ↓
5. /forge:security-review       — аудит безопасности
   ↓
6. /forge:validate              — проверка соответствия плану
   ↓
7. /forge:sync                  — обновление документации
   ↓
8. (Pull Request / Merge)
```

### API разработка

```
1. /forge:brainstorm            — исследование требований API
   ↓
2. /forge:api-design            — проектирование контракта
   ↓
3. /forge:write-plan            — план реализации
   ↓
4. /forge:execute-plan          — реализация эндпоинтов
```

### Миграция базы данных

```
1. /forge:brainstorm            — анализ требуемых изменений
   ↓
2. /forge:migrate               — план миграции + rollback
   ↓
3. /forge:write-plan            — план изменений в коде
   ↓
4. /forge:execute-plan          — реализация
```

### Деплой и релиз

```
1. /forge:deploy                — настройка деплоймента
   ↓
2. /forge:security-review       — аудит безопасности
   ↓
3. /forge:validate              — финальная проверка
```

### Быстрая разработка (известный проект)

```
1. /forge:brainstorm            — если нужен дизайн
   ↓
2. /forge:write-plan            — план
   ↓
3. /forge:execute-plan          — реализация
   ↓
4. /forge:sync                  — синхронизация
   ↓
5. /forge:validate              — проверка
```

### Только документирование (существующий код)

```
1. /forge:init                  — если нет .forge/
   ↓
2. /forge:sync                  — синхронизация
   ↓
3. /forge:validate              — проверка полноты
```

## Команды по этапам разработки

### Инициализация
- `/forge:init` — один раз для нового проекта (5 параллельных субагентов, YAML формат)
- `/forge:graph` — графовая карта кода (автоматически при init, если graphify установлен)

### Планирование
- `/forge:brainstorm` — исследование подхода
- `/forge:write-plan` — детальный план
- `/forge:api-design` — проектирование API контракта

## Частые ошибки

**Пропускаете `/forge:init`**
- Результат: нет контекстной документации
- Решение: Запустите `/forge:init` в начале

**Забываете `/forge:sync`**
- Результат: документация отстаёт от кода
- Решение: Запускайте `/forge:sync` после каждой фичи

**Используете `/forge:execute-plan` без плана**
- Результат: команда не запустится
- Решение: Сначала `/forge:write-plan`, потом `/forge:execute-plan`

**Пропускаете `/forge:security-review` перед деплоем**
- Результат: уязвимости попадают в продакшн
- Решение: Всегда `/forge:security-review` перед `/forge:deploy`

**Меняете схему БД без `/forge:migrate`**
- Результат: простой, потеря данных, нет rollback
- Решение: Всегда `/forge:migrate` для изменений схемы

## Tips & Tricks

**Используйте `/forge:sync` часто**
- После каждой законченной задачи
- Перед созданием PR
- При переключении между фичами

**Используйте `/forge:validate` как проверку качества**
- Убедитесь что план реализован полностью
- Проверьте что документация актуальна
- Выявите проблемы на ранних этапах

**Читайте `.forge/index.yml` перед началом работы**
- Там текущая задача и прогресс
- Список отложенных дел
- История из предыдущих сессий

**Смотрите `.forge/map.yml` чтобы понять проект**
- Структура директорий
- Red zone файлы (критичные)
- Количество файлов в каждой папке

**Комбинируйте `/forge:api-design` + `/forge:migrate`**
- Сначала API контракт, потом схема БД под него
- Обеспечивает согласованность data layer и API layer

**`/forge:security-review` как гейт перед мёржем**
- Добавьте в workflow перед каждым PR в main
- FAIL = блокирует мёрж до исправления
