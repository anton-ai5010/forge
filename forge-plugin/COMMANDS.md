# FORGE Plugin Commands Reference

Полный справочник команд forge-plugin (30+ скиллов под капотом) с описанием назначения и использования.

Большинство команд — это тонкая обёртка над одноимённым скиллом (`/forge:new-task` → skill `new-task`, `/forge:plan` → skill `plan`, и т.д.). Команда вызывает скилл, скилл делает работу. Так логику можно править в одном месте.

---

## Сердце плагина: 6-фазный pipeline (0 → 1 → 1.5 → 2 → 3 → 4)

Главные команды — это конвейер от сырой мысли пользователя до закоммиченного кода. Phase 0 — навигатор **над** пайплайном (запускается, когда непонятно КУДА двигать проект); дальше фазы идут подряд:

```
/forge:unblocker → /forge:new-task → /forge:refine-idea → /forge:plan → /forge:critique → /forge:execute
    Phase 0           Phase 1            Phase 1.5          Phase 2         Phase 3          Phase 4
  Направление        Понимание         Разбор идеи           План           Критика         Реализация
```

**Что делает каждая фаза:**

| Фаза | Команда | Что на входе | Что на выходе |
|------|---------|--------------|---------------|
| 0. Направление | `/forge:unblocker` | непонятно куда двигать / застрял | карта проекта + направления, выбранный шаг → в `/forge:new-task` |
| 1. Понимание | `/forge:new-task` | сырой промпт | чистая задача + критерий готовности |
| 1.5. Разбор идеи | `/forge:refine-idea` | чистая задача | та же задача, доработанная после реалити-чека идеи |
| 2. Планирование | `/forge:plan` | доработанная задача | `.forge/plans/YYYY-MM-DD-<slug>.md` — план с чекпоинтами |
| 3. Критика | `/forge:critique` | план | план + Execution Strategy (или возврат в Phase 2) |
| 4. Реализация | `/forge:execute` | укреплённый план | реализованный код, коммиты |

**Auto-handoff на "ОК":**
Когда фаза заканчивается, Claude показывает результат и ждёт. Если пользователь отвечает короткое **"ОК"** / **"го"** / **"дальше"** — следующая фаза стартует автоматически без явного `/forge:plan`. Если пользователь хочет правок — он их пишет, и фаза доделывается. Это позволяет одним коротким "ок" прокатить весь pipeline.

**Условный handoff Phase 3 → Phase 4 (confidence-гейт):**
Критика прогоняет план через 4 персоны (Скептик, Прагматик, Архитектор, Адвокат пользователя). Каждая персона помечает свои находки **confidence-числом 0–100**. В план попадают только находки с **confidence ≥ 80%**. После этого:
- Если фатальных дыр **не найдено** (≥80%) → план дополняется Execution Strategy и автоматически передаётся в `/forge:execute`.
- Если **есть фатальная дыра** (≥80% и блокирует реализацию) → handoff в `/forge:execute` НЕ происходит, возврат в `/forge:plan` на доработку.

Это значит: нельзя случайно выйти из критики в реализацию с дырявым планом. Гейт работает автоматически.

**Когда фазы можно пропустить:**
- `/forge:new-task` пропускается, если задача уже сформулирована чисто и есть критерий.
- `/forge:critique` пропускать **нельзя** — именно когда план "выглядит ок", критика нужнее всего (планировщик влюблён в свой план).

**Автоматический GitHub-sync (опционально):**
Если включён флаг `github_sync: true` в `.forge/index.yml` (или плагин сам предложил при первой задаче через `sync.sh should-offer`), каждая фаза автоматически публикует артефакты на GitHub:
- `/forge:new-task` → Issue с задачей и критерием готовности
- `/forge:critique` → комментарий с результатом критики + Sub-issues под каждый шаг финального плана (черновик из `/forge:plan` на GitHub не попадает)
- `/forge:execute` → закрытие Sub-issues по мере прохождения шагов

Pinned Issue (карта целей со счётчиками) и шапка `README.md` пересобираются механически внутри sync.sh (create-task, add-steps) и в финале execute.

**Память проекта в git (всегда, независимо от github_sync):**
`.forge/` (задачи, планы, идеи, решения, журнал) — часть репозитория, а не только локальная папка. Итог сессии и мерж автоматически коммитят память и пушат ветку (скилл `memory-backup`); служебный мусор отсечён через `.forge/.gitignore`. Ручной запуск — `/forge:save` или слово «сохрани». Если память не сохранялась больше суток — плагин напомнит при старте сессии. У проекта без удалёнки плагин предложит создать приватный репозиторий (только с явного «да»). Секреты в `.forge` не пишутся — только названия записей Bitwarden. Projects v2 board не используется (см. `.forge/decisions.yml: no-projects-v2-board`).

Управление картой целей — через `/forge:roadmap`.

## 0. /forge:start

**Назначение:** Полная загрузка контекста проекта в начале сессии

**Когда использовать:**
- Один раз при открытии проекта в новой сессии
- Когда нужен полный дашборд текущего состояния

**Что делает:**
- Через backtick-препроцессинг мгновенно загружает: index.yml, status.yml, decisions.yml, dead-ends, journal.yml, git log
- Показывает дашборд: цель, стадия, что работает/сломано, провальные подходы
- Предлагает продолжить незаконченную задачу

**Результат:** Claude полностью в контексте, готов работать без дополнительных вопросов

**Примечание:** Между запросами контекст поддерживается автоматически через UserPromptSubmit hook (инжект index.yml + dead-ends). `/forge:start` нужен только в начале.

## 1. /forge:init

**Назначение:** Инициализация структуры FORGE документации в проекте

**Когда использовать:**
- Один раз в начале разработки проекта
- При миграции старого проекта на FORGE
- Для создания контекстной документации

**Что делает:**
- Запускает 5 параллельных субагентов для сканирования проекта:
  - Структура кода и критичные файлы (red zones)
  - Конвенции проекта (язык, именование, стиль)
  - Инфраструктура (Docker, базы данных, серверы, CI/CD)
  - Продуктовый и технический контекст
  - Графовая карта кода (если graphify установлен)
- Создаёт `.forge/` директорию с:
  - `map.yml` — структура и критичные файлы (red zones)
  - `conventions.yml` — соглашения проекта (язык, именование)
  - `index.yml` — точка входа: цель, стадия, сессия, каталог ресурсов
  - `infrastructure.yml` — Docker, БД, серверы, CI/CD конфигурация
  - `status.yml`, `decisions.yml`, `dead-ends.yml`, `journal.yml` — память проекта (что работает, решения, тупики, история сессий)
  - `direction.yml` — стратегический слой (направления к цели, для навигатора Phase 0)
  - `structure.md` — карта директорий человекочитаемо
  - `library/` — зеркало документации под структуру проекта
- Генерирует всё в YAML формате

**Результат:** Полная структура контекстной документации (~2k токенов вместо ~40k+ исходного кода)

**Пример:**
```
/forge:init

Инициализировать футбольный предиктор.
Язык: Python
Red zones: indicators/smart_money.py, strategies/production.py
```

## 2. /forge:sync

**Назначение:** Обновление документации после внесения изменений в код

**Когда использовать:**
- После каждой реализованной задачи/фичи
- Перед коммитом
- Перед PR в основную ветку
- Для поддержания актуальности документации

**Что делает:**
- Определяет изменения со времени последней синхронизации (git diff)
- Обновляет `.forge/library/` для новых/изменённых файлов
- Обновляет счёты файлов в `.forge/map.yml`
- Обновляет `.forge/index.yml` со стадией и последней сессией
- Проверяет и обновляет `.forge/infrastructure.yml` (Docker, БД, серверы)
- Добавляет запись в `.forge/journal.yml`

**Результат:** Актуальная документация, готовая для следующей итерации

**Пример:**
```
/forge:sync

Я завершил Phase 9 Web Interface.
Добавлены: api/, scripts/run_server.py, тесты
Обновлены: PredictionPipeline, test suites
```

## 3. /forge:new-task (Phase 1: Understanding)

**Назначение:** Превратить сырой промпт в чистую задачу с критерием готовности

**Под капотом:** инвокает skill `new-task`

**Когда использовать:**
- В самом начале любой новой задачи
- Когда формулировка размыта или содержит несколько целей сразу
- Перед тем как переходить к планированию

**Что делает:**
- Парсит сырой ввод и отделяет суть от шума
- Логические/продуктовые вопросы — задаёт пользователю
- Технические детали — Claude разбирает сам (читает код, .forge/)
- При желании рисует HTML-набросок для подтверждения
- Формулирует чистую задачу + явный критерий "готово"

**Handoff:** после подтверждения задачи сразу инвокает `/forge:refine-idea` (Phase 1.5) — разбор идеи доработает задачу и сам передаст её в `/forge:plan`

**Результат:** Чистая постановка задачи и критерий приёмки в `.forge/tasks/YYYY-MM-DD-<slug>.md`, передаются в `/forge:refine-idea`

**Пример:**
```
/forge:new-task

Хочу чтобы корзина не сбрасывалась при F5 и работало на мобилке.
```

## 3.5. /forge:refine-idea (Phase 1.5: Idea Check)

**Назначение:** Реалити-чек самой идеи до плана — та ли проблема, нет ли пути проще, стоит ли усилий

**Под капотом:** инвокает skill `refine-idea`

**Когда использовать:**
- Автоматически после `/forge:new-task` — Phase 1 сама инвокает эту фазу, ждать явной команды не нужно
- Вручную: "разбери идею", "покритикуй идею", "а это вообще разумно?"

**Что делает:**
- Читает задачу из `.forge/tasks/YYYY-MM-DD-<slug>.md`
- Тихо изучает контекст проекта: decisions, dead-ends, status, существующий код
- Зеркалит как понял идею и оспаривает её фактами проекта: та ли проблема, нет ли пути проще, что сломает, стоит ли усилий
- На каждую слабость даёт конкретную альтернативу, а не приговор
- Это диалог, не отчёт: один живой проход в основной сессии, не 4 персоны (персоны — это `/forge:critique`, и они рвут план, а не идею)

**Handoff:** когда сошлись на доработанной идее — сразу инвокает `/forge:plan` (Phase 2)

**Результат:** Доработанный `.forge/tasks/YYYY-MM-DD-<slug>.md` (при существенных изменениях — секция `## Доработано на разборе`)

**Пример:**
```
/forge:refine-idea

Разбери идею из задачи cart-persist: точно ли нужна своя корзина, а не готовое решение?
```

## 4. /forge:plan (Phase 2: Planning)

**Назначение:** Построить план с явными чекпоинтами под чистую задачу

**Под капотом:** инвокает skill `plan`

**Когда использовать:**
- После `/forge:refine-idea` (задача прошла реалити-чек идеи) — или после `/forge:new-task`, если разбор идеи пропущен
- Перед реализацией любой нетривиальной работы

**Что делает:**
- Раскладывает задачу на шаги с чекпоинтами (точки остановки и проверки — на смысловых вехах, а не каждые N шагов)
- Технику разбирает сам (читает код, библиотеки), у пользователя спрашивает только бизнес-логику
- Рекурсивно отдаёт дальние блокеры в отдельные сессии — чтобы основной контекст не засорять
- Привязывает каждый шаг к файлам и проверкам
- Сохраняет план в `.forge/plans/`

**Handoff:** на коротком "ОК" автоматически передаёт управление в `/forge:critique`

**Результат:** Файл `.forge/plans/YYYY-MM-DD-<slug>.md` с шагами и чекпоинтами

**Пример:**
```
/forge:plan

Использовать чистую задачу из /forge:new-task: персистентная корзина + мобильная вёрстка.
```

## 5. /forge:critique (Phase 3: Critique)

**Назначение:** Прогнать план через 4 параллельных персоны-критика и дописать Execution Strategy

**Под капотом:** инвокает skill `critique`

**Когда использовать:**
- Сразу после `/forge:plan`, до начала реализации
- Когда план "выглядит ок" — именно тогда критика нужнее всего (планировщик влюблён в свой план)

**Что делает:**
- Запускает 4 параллельных персоны: **Скептик**, **Прагматик** (YAGNI), **Архитектор**, **Адвокат пользователя**
- Каждая персона помечает свои находки **confidence-числом 0–100**
- В план попадают только находки с **confidence ≥ 80%** — остальное отбрасывается как "может быть"
- Сводит претензии и дописывает в план секцию **Execution Strategy** (что в субагентов, что параллельно, что последовательно)

**Conditional handoff в /forge:execute:**
- Нет фатальных дыр (≥80%) → автоматический переход в `/forge:execute` на коротком "ОК"
- Есть фатальная дыра (≥80% и блокирует реализацию) → handoff в execute НЕ происходит, возврат в `/forge:plan` на доработку

**Результат:** План с пометками критиков + Execution Strategy, готовый к исполнению

**Пример:**
```
/forge:critique

Прокритиковать план .forge/plans/2026-05-13-cart-persist.md
```

## 6. /forge:execute (Phase 4: Implementation)

**Назначение:** Реализовать план — грязную работу в субагентах, остановка на чекпоинтах

**Под капотом:** инвокает skill `execute`

**Когда использовать:**
- После `/forge:critique` (план укреплён, фатальных дыр ≥80% нет)
- Когда нужна пошаговая реализация с верификацией

**Что делает:**
- Идёт по плану шаг за шагом в основной сессии
- Тяжёлую работу (чтение пачки файлов, длинные тесты, разбор логов) делегирует субагентам — их изолированный контекст возвращает короткое summary, основной контекст остаётся чистым
- Останавливается только на чекпоинтах плана (на смысловых вехах), а не каждые N шагов
- После одобрения чекпоинта — коммитит и идёт дальше

**Handoff:** при достижении критерия готовности — передаёт в `/forge:validate` / `/forge:sync`

**Результат:** Реализованный план, все коммиты, готовый код

**Пример:**
```
/forge:execute

Выполнить план .forge/plans/2026-05-13-cart-persist.md
```

## 7. /forge:discover

**Назначение:** Поиск готовых плагинов в маркетплейсе (для ускорения разработки)

**Когда использовать:**
- После написания плана, перед выбором подхода к реализации
- Когда нужны готовые инструменты для ускорения
- Для проверки, существует ли уже то, что нужно

**Что делает:**
- Анализирует план
- Извлекает технологии, домены, паттерны
- Ищет релевантные плагины в маркетплейсе
- Рекомендует высоко рейтинговые решения
- Предлагает установить или создать кастомный плагин

**Результат:** Список рекомендованных плагинов или перечень того, что нужно создать

**Примечание:** Маркетплейс API ещё в разработке. Сейчас команда показывает, что БЫ было при наличии API.

## 8. /forge:validate

**Назначение:** Проверка соответствия кода плану и документации (read-only)

**Когда использовать:**
- Перед мёржем фичи в основную ветку
- После завершения реализации плана
- Когда что-то кажется несоответствующим
- Для верификации перед PR

**Что делает:**
- Сравнивает код с планом (все ли задачи завершены?)
- Проверяет документацию против кода (все ли файлы задокументированы?)
- Проверяет точность `.forge/map.yml` (счёты файлов совпадают?)
- Выявляет несоответствия и предлагает действия

**Результат:** Подробный отчёт о несоответствиях (если есть) с рекомендациями по исправлению

**Примечание:** Только чтение и анализ, не модифицирует файлы

**Пример:**
```
/forge:validate

Проверить что Phase 9 полностью реализована по плану
```

## 9. /forge:cleanup

**Назначение:** Анализ качества кода по всему проекту с параллельными агентами

**Когда использовать:**
- Когда код стал запутанным после быстрого прототипирования
- Перед релизом — навести порядок
- Для погашения технического долга
- Периодически для поддержания качества

**Что делает:**
- Разбивает проект на независимые области (по директориям/модулям)
- Запускает параллельных агентов для анализа каждой области
- Проверяет: мёртвый код, именование, сложность, дублирование, структуру, конвенции
- Собирает сводный отчёт по severity (high/medium/low)
- Спрашивает что исправить (всё / только critical / выборочно)
- Исправляет только после одобрения

**Результат:** Структурированный отчёт + исправления по запросу

**Примечание:** Только анализ до одобрения, red zone файлы никогда не правятся автоматически

**Пример:**
```
/forge:cleanup

Проверить качество кода во всём проекте.
Особое внимание: именование, дублирование, мёртвый код.
```

## 10. /forge:unblocker (Phase 0: Direction)

**Назначение:** Навигатор направления — понять КУДА двигать проект, когда непонятно или застрял

**Когда использовать:**
- "Что дальше по проекту", "куда двигаемся", "направление"
- "Застрял", "не знаю что делать", "с чего начать"
- Жалобы на проект: "не нравится", "ну такое"

**Что делает:**
- Показывает живую цель проекта (якорь из `index.yml`)
- Читает всю `.forge`-память + сканирует код
- Рисует карту проекта: все части механизма со статусом (честно, без завышения)
- Расписывает ВСЕ направления к цели кратко, рекомендует дефолт-движение — но финальный выбор за пользователем
- Заканчивает одним физическим шагом

**Результат:** обновлённые `.forge/direction.yml` (память для Клода) + `ROADMAP.md` (человекочитаемо) + карта проекта

**Handoff:** выбранный первый шаг подаётся в `/forge:new-task` (Phase 1)

**Важно:** Это фаза НАПРАВЛЕНИЯ — код не пишется, баги не чинятся.

**Пример:**
```
/forge:unblocker

Застрял с этим проектом, не знаю за что браться.
```

## 11. /forge:design

**Назначение:** Генерация UI/UX дизайн-системы для проекта

**Когда использовать:**
- При создании нового фронтенд-проекта или редизайне
- Когда нужна согласованная визуальная система

## 12. /forge:security-review

**Назначение:** Проверка безопасности проекта по чеклисту OWASP

**Когда использовать:**
- Перед деплоем в продакшн
- После добавления аутентификации, API, обработки данных
- Периодически для аудита безопасности
- Перед мёржем крупных фич

**Что делает:**
- Сканирует проект по 6 категориям:
  - **Secrets** — захардкоженные ключи, токены, пароли в коде и конфигах
  - **Input Validation** — SQL injection, XSS, path traversal, command injection
  - **Auth** — аутентификация, авторизация, управление сессиями
  - **API Security** — rate limiting, CORS, валидация запросов, error leaking
  - **Dependencies** — уязвимые зависимости, устаревшие пакеты
  - **Data Privacy** — PII в логах, шифрование, GDPR compliance
- Для каждой категории выдаёт PASS / FAIL с деталями

**Результат:** Структурированный отчёт PASS/FAIL по каждой категории с конкретными находками и рекомендациями

**Пример:**
```
/forge:security-review

Полный аудит безопасности перед релизом v2.0.
Особое внимание: API endpoints, обработка пользовательских данных.
```

## 14. /forge:migrate

**Назначение:** Планирование миграций базы данных с zero-downtime стратегией

**Когда использовать:**
- При изменении схемы БД (новые таблицы, колонки, индексы)
- При breaking changes в структуре данных
- Когда нужна миграция без простоя

**Что делает:**
- Анализирует текущую схему БД и требуемые изменения
- Для breaking changes применяет expand-contract паттерн:
  1. **Expand** — добавить новое, не удаляя старое
  2. **Migrate data** — перенести данные в новую структуру
  3. **Contract** — удалить старое после полного перехода
- Генерирует forward и rollback скрипты (rollback обязателен)
- Проверяет совместимость с текущим кодом
- Оценивает время и риски миграции

**Результат:** Файл `.forge/plans/migration-{название}.yml` с планом миграции, SQL скриптами и rollback

**Пример:**
```
/forge:migrate

Разделить таблицу users на users + user_profiles.
Текущая БД: PostgreSQL, ~2M записей.
Требование: zero-downtime, обратная совместимость на время перехода.
```

## 16. /forge:graph

**Назначение:** Построить или обновить графовую карту кода (зависимости, импорты, вызовы)

**Когда использовать:**
- После `/forge:init` для большого проекта (50+ файлов)
- Когда нужно понять архитектуру и связи между модулями
- Перед `/forge:new-task` / debugging для быстрой навигации по коду

- Генерирует отчёт `.forge/graph-report.md`

**Режимы:**
- `/forge:graph` — сборка/обновление графа (tree-sitter, без LLM)

- `/forge:graph --query "вопрос"` — поиск по графу
- `/forge:graph --path "A" "B"` — путь между компонентами
- `/forge:graph --explain "Node"` — объяснение узла и его соседей

**Требования:** graphify (`pipx install graphifyy`)

**Автоматизация:** строится при `/forge:init`, обновляется при `/forge:sync`

**Пример:**
```
/forge:graph
/forge:graph --query "как работает авторизация?"
/forge:graph --path "UserModel" "PaymentService"
```

## 21. /forge:hookify

**Назначение:** Превратить повторяющееся "Антон поправил Claude" в постоянное правило-хук

**Под капотом:** инвокает skill `hookify`

**Когда использовать:**
- Когда поймал себя на том, что одно и то же повторяешь Claude по 3-й раз в сессии

- Когда есть явное правило, которое хочется зафиксировать ("всегда отвечай по-русски", "не предлагай тесты пока я не попросил")

**Что делает:**
- **Без аргументов:** сканит историю текущей сессии, находит места где пользователь повторно исправлял Claude по одной и той же теме, предлагает правило

**Результат:** Файл `.forge/hookrules/{название}.md` с правилом — хук подхватит его уже на следующем вызове Edit/Write, перезапуск сессии не нужен

**Пример:**
```
/forge:hookify

/forge:hookify Никогда не добавляй комментарии в коде если я не попросил.
```

## Workflows

### Полный цикл разработки (6-фазный пайплайн)

```
1. /forge:init                  — инициализация проекта
   ↓
2. /forge:unblocker             — Phase 0: Direction (если непонятно куда двигать; иначе пропускается)
   ↓
3. /forge:new-task              — Phase 1: Understanding (чистая задача + критерий)
   ↓
4. /forge:refine-idea           — Phase 1.5: Idea Check (реалити-чек идеи; new-task инвокает сам)
   ↓
5. /forge:plan                  — Phase 2: Planning (план с чекпоинтами)
   ↓
6. /forge:critique              — Phase 3: Critique (4 персоны + Execution Strategy)
   ↓
7. /forge:execute               — Phase 4: Implementation (грязная работа в субагентах, стоп на чекпоинтах)
   ↓
8. /forge:security-review       — аудит безопасности
   ↓
9. /forge:validate              — проверка соответствия плану
   ↓
10. /forge:sync                 — обновление документации
   ↓
11. (Pull Request / Merge)
```

### API разработка

```
1. /forge:new-task              — Phase 1: чистая задача API
   ↓
2. /forge:refine-idea           — Phase 1.5: реалити-чек идеи
   ↓
3. /forge:api-design            — проектирование контракта
   ↓
4. /forge:plan                  — Phase 2: план реализации
   ↓
5. /forge:critique              — Phase 3: критика плана
   ↓
6. /forge:execute               — Phase 4: реализация эндпоинтов
```

### Миграция базы данных

```
1. /forge:new-task              — Phase 1: чистая задача миграции
   ↓
2. /forge:refine-idea           — Phase 1.5: реалити-чек идеи
   ↓
3. /forge:migrate               — план миграции + rollback
   ↓
4. /forge:plan                  — Phase 2: план изменений в коде
   ↓
5. /forge:critique              — Phase 3: критика (риски данных)
   ↓
6. /forge:execute               — Phase 4: реализация

```

   ↓

   ↓

   ↓

   ↓

## Команды по этапам разработки

### Инициализация
- `/forge:init` — один раз для нового проекта (5 параллельных субагентов, YAML формат)
- `/forge:graph` — графовая карта кода (автоматически при init, если graphify установлен)

### Направление (Phase 0)
- `/forge:unblocker` — навигатор: карта проекта + все направления, выбранный шаг → в `/forge:new-task`

### Понимание задачи (Phase 1)
- `/forge:new-task` — сырой промпт → чистая задача + критерий готовности

### Разбор идеи (Phase 1.5)
- `/forge:refine-idea` — реалити-чек идеи фактами проекта до плана (new-task инвокает сам)

### Планирование (Phase 2)
- `/forge:plan` — план с чекпоинтами, рекурсия на дальние блокеры
- `/forge:api-design` — проектирование API контракта
- `/forge:migrate` — планирование миграции БД
- `/forge:discover` — поиск готовых решений

### Критика (Phase 3)
- `/forge:critique` — 4 параллельных персоны рвут план + дописывают Execution Strategy

### Дизайн
- `/forge:design` — UI/UX дизайн-система (BM25 поиск по CSV базам)

### Реализация (Phase 4)
- `/forge:execute` — реализация плана, грязная работа в субагентах, стоп на чекпоинтах
- `/forge:deploy` — настройка деплоймента и CI/CD
- `/forge:sync` — обновление документации после работы

### Верификация
- `/forge:validate` — проверка соответствия плану и .forge docs
- `/forge:security-review` — OWASP аудит безопасности (PASS/FAIL)
- `/forge:cleanup` — анализ качества кода и рефакторинг
- `/forge:sync` — финальная синхронизация перед PR

### Помощь
- `/forge:unblocker` — разблокировка когда застрял
- `/forge:investigate` — разобраться что происходит ДО починки
- `/forge:explain` — визуальный ответ "как работает X?" в HTML
- `/forge:product-map` — навигатор проекта в HTML

### Эволюция плагина под пользователя
- `/forge:hookify` — превратить повторение в правило-хук
- `/forge:evolve` — кластеризация сквозных болей + предложения автоматизации
- `/forge:session-insights` — паттерны работы из истории сессий

## Частые ошибки

**Пропускаете `/forge:init`**
- Результат: нет контекстной документации
- Решение: Запустите `/forge:init` в начале

**Забываете `/forge:sync`**
- Результат: документация отстаёт от кода
- Решение: Запускайте `/forge:sync` после каждой фичи

**Используете `/forge:execute` без плана**
- Результат: команда не запустится
- Решение: Пройдите фазы — `/forge:new-task` → `/forge:refine-idea` → `/forge:plan` → `/forge:critique` → `/forge:execute`

**Пропускаете `/forge:critique` перед `/forge:execute`**
- Результат: план реализуется с неучтёнными рисками и дырами
- Решение: Всегда прогоняйте план через `/forge:critique` — именно когда план "выглядит ок"

**Пропускаете `/forge:security-review` перед деплоем**
- Результат: уязвимости попадают в продакшн
- Решение: Всегда `/forge:security-review` перед `/forge:deploy`

**Меняете схему БД без `/forge:migrate`**
- Результат: простой, потеря данных, нет rollback
- Решение: Всегда `/forge:migrate` для изменений схемы

## Tips & Tricks

**Используйте `/forge:sync` часто**
- После каждой законченной задачи
- Перед созданием PR
- При переключении между фичами

**Используйте `/forge:validate` как проверку качества**
- Убедитесь что план реализован полностью
- Проверьте что документация актуальна
- Выявите проблемы на ранних этапах

**`.forge/index.yml` автоматически инжектится в контекст каждого промпта — вручную читать не нужно**
- Там текущая задача и прогресс
- Список отложенных дел
- История из предыдущих сессий

**Смотрите `.forge/map.yml` чтобы понять проект**
- Структура директорий
- Red zone файлы (критичные)
- Количество файлов в каждой папке

**Комбинируйте `/forge:api-design` + `/forge:migrate`**
- Сначала API контракт, потом схема БД под него
- Обеспечивает согласованность data layer и API layer

**`/forge:security-review` как гейт перед мёржем**
- Добавьте в workflow перед каждым PR в main
- FAIL = блокирует мёрж до исправления

## Быстрая помощь

```bash
# Инициализировать новый проект
/forge:init

# Понять куда двигать проект (если застрял)
/forge:unblocker        # Phase 0: Direction

# Спланировать новую фичу (6-фазный пайплайн)
/forge:new-task         # Phase 1: Understanding
/forge:refine-idea      # Phase 1.5: Idea Check (new-task инвокает сам)
/forge:plan             # Phase 2: Planning
/forge:critique         # Phase 3: Critique

# Реализовать план
/forge:execute          # Phase 4: Implementation

# Сохранить память проекта (.forge) в git и на GitHub
/forge:save             # или просто слово «сохрани»

# Спроектировать API
/forge:api-design

# Спланировать миграцию БД
/forge:migrate

# Настроить деплой
/forge:deploy

# Создать дизайн-систему
/forge:design

# Проверить безопасность
/forge:security-review

# Построить графовую карту кода
/forge:graph

# Навигатор проекта (HTML)
/forge:product-map

# Визуальное объяснение "как работает X?"
/forge:explain "как работает оплата?"

# Синхронизировать документацию
/forge:sync

# Проверить что всё сделано правильно
/forge:validate

# Зафиксировать правило из повторений в сессии
/forge:hookify

# Раз в 2 недели — кластеризация сквозных болей
/forge:evolve
```
