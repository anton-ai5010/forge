# Execute — Phase 4: Doing it without polluting context

Вход: финальный план из фазы 3 (`.forge/plans/...`) с секцией Execution Strategy.
Выход: реализованная задача согласно критерию готовности.

## Цель фазы

Реализовать план **в той же сессии**, но **с чистым контекстом**. Это достигается за счёт того что грязная работа делается в субагентах — их контекст изолирован, в основную сессию возвращается только короткий итог.

**Почему это критично:** длинный автономный прогон в основной сессии = деградация качества к концу (старое сжимается, точность падает, ошибки компаундятся). Делегирование тяжёлого в субагентов = основная сессия остаётся свежей и может принимать решения / общаться с тобой.

## Главный принцип: что в субагент, что в основную сессию

### В субагент (грязная работа)
- Чтение 5+ файлов с целью что-то найти → возвращает короткий итог
- Запуск длинных тестов / билдов → возвращает PASS/FAIL + что упало
- Анализ логов / больших дампов → возвращает выводы
- Поиск по проекту / гугление → возвращает релевантное
- Массовая правка похожих файлов → возвращает diff-сводку

### В основной сессии (мышление и связь с пользователем)
- Решения / выбор подхода
- Финальная стыковка кусков
- Диалог с пользователем на чекпоинтах
- Простая правка одного-двух файлов
- Реакция на проблемы / уточнения у пользователя

**Правило большого пальца:** если шаг производит много токенов вывода — в субагент. Если шаг короткий и требует контекста разговора — в основной.

## Process state (для statusline)

В начале фазы:

```bash
mkdir -p .forge && cat > .forge/state.yml <<EOF
phase: execute
task: <slug-из-плана>
started_at: $(date -Iseconds)
EOF
```

При **финальном завершении задачи** (после прохождения критерия готовности) — очисти state:

```bash
echo "phase: idle" > .forge/state.yml
```

## Процесс

### 1. Прочитай финальный план и задачу

- `.forge/plans/YYYY-MM-DD-<slug>.md` — план с Execution Strategy.
- `.forge/tasks/YYYY-MM-DD-<slug>.md` — задача + критерий готовности (для контроля что выполнили).
- Если плана нет — спроси: *"План не найден. Делал ли `/plan` и `/critique`?"*
- Если нет секции Execution Strategy — спроси: *"План есть, но без стратегии выполнения. Делал ли `/critique`?"*

### 1.5. Обеспечь рабочую ветку под задачу

Перед первым изменением кода убедись, что работа пойдёт в отдельной ветке под эту задачу, а не в основную. Делается **молча**, если ситуация однозначна — пользователь (часто не-кодер) не должен думать про git.

`slug` — это имя файла плана, который ты прочитал на шаге 1, без даты и `.md` (например `.forge/plans/2026-06-25-auto-branch-per-task.md` → `auto-branch-per-task`). НЕ сканируй `.forge/plans/` заново — бери тот файл, по которому идёт выполнение.

Определи основную ветку и текущее состояние:

```bash
BASE=$(git symbolic-ref --short refs/remotes/origin/HEAD 2>/dev/null | sed 's@^origin/@@')
if [ -z "$BASE" ]; then
  git show-ref --verify --quiet refs/heads/main && BASE=main || BASE=master
fi
CUR=$(git branch --show-current)   # пусто при detached HEAD
```

Дальше разбери случаи по порядку (первый подошедший — выполняй, остальные пропусти):

1. **Уже в нужной ветке или worktree** — `CUR` == `feat/<slug>` (ровно эта задача), ИЛИ ты внутри связанного git-worktree (проверка: `[ "$(git rev-parse --git-common-dir)" != "$(git rev-parse --git-dir)" ]` истинно). → **Ничего не делай, молча продолжай.** (Закрывает повторный запуск на своей ветке и worktree-сценарий.)
   **ВАЖНО:** любая ДРУГАЯ ветка с префиксом `feat/` / `fix/`, не равная `feat/<slug>`, сюда НЕ попадает — она уходит в пункт 5. Иначе новая задача молча намешается в чужую недоделку (ровно то, от чего защищает пункт 5).

2. **Ветка задачи уже существует** — `git show-ref --verify --quiet refs/heads/feat/<slug>` истинно. → `git checkout feat/<slug>` (переключись, НЕ создавай через `-b`). Закрывает повторный запуск execute.

3. **Ты на основной ветке** — `CUR` == `$BASE`. → `git checkout -b feat/<slug>`. Git сам перенесёт незакоммиченные правки на новую ветку — stash не нужен. **Без объявления «ветка создана»** — молча, пользователь не просил это видеть.

4. **Detached HEAD** — `CUR` пустой. → НЕ предлагай «поверх текущей». Одной фразой: *«Ты сейчас не в рабочей зоне. Начинаю задачу с чистого листа, ок?»* → если есть незакоммиченные правки, сначала `git stash -u` (потом скажи человеку: *«Твою работу отложил в сохранность»*), затем `git checkout -b feat/<slug> "$BASE"`.

5. **Ты на чужой ветке** — `CUR` != `$BASE` и пункты 1–2 не подошли. → **НЕ создавай ветку молча.** Спроси на языке смысла, без git-жаргона (не говори «слита», «master», «поверх»):
   > *«Ты сейчас в другой незавершённой работе — "<название задачи той ветки, если найдёшь по имени плана/задачи>". Новая задача — это её продолжение или отдельное дело? (по умолчанию — отдельное, начну с чистого листа)»*
   - **«отдельное» / по умолчанию / просто повторяет «делай»** → если дерево грязное: `git stash -u` → `git checkout "$BASE"` → `git checkout -b feat/<slug>`, затем скажи человеку: *«Твою незавершённую работу отложил в сохранность — вернёшься к ней, всё на месте.»* Если дерево чистое — просто переключись на `$BASE` и создай ветку.
   - **«продолжение»** → `git checkout -b feat/<slug>` от текущей ветки.

**Принцип:** forge НЕ угадывает связанность задачи с веткой молча — пункт 5 всегда спрашивает. Дефолт — отдельная задача от основной ветки, чтобы случайно не намешать новое в чужую недоделку.

### 2. Создай TodoWrite по шагам плана

Каждый шаг плана → одна todo. Пометь у каждой:
- `[main]` — делать в основной сессии
- `[agent]` — делегировать субагенту
- `[parallel-with-N]` — параллельно с другими помеченными такой же группой

### 3. Выполняй шаги по плану

Иди по списку. Для каждого шага:

#### Если `[main]`:
- Делай сам в основной сессии
- TodoWrite: in_progress → completed по завершении
- Не показывай каждый под-шаг пользователю (он засыпает) — показывай результат когда шаг завершён

#### Если `[agent]`:
- Запусти субагента через Agent tool
- В промпт субагенту положи:
  - Что конкретно сделать (из плана)
  - Какие файлы трогать
  - Что вернуть в основной сессию (короткий формат — diff/итог/PASS-FAIL, не дамп логов)
- Жди ответ
- Применяй результат в основной сессии

#### Если `[parallel-with-N]`:
- Запусти все шаги группы **одним сообщением** через несколько Agent tool вызовов
- Жди все ответы
- Применяй

#### После каждого выполненного шага — GitHub-sync (тихо если выключен)

```bash
bash $CLAUDE_PLUGIN_ROOT/skills/github-sync/sync.sh close-step <task-slug> <step-num>
```

Закроет соответствующий sub-issue на GitHub. Молча no-op если sync выключен или sub-issues не создавались.

### 4. На чекпоинте — стоп и показ

Чекпоинты в плане размечены явно (например `### ✅ Чекпоинт A`).

На чекпоинте:
1. Останови выполнение
2. Покажи пользователю **что готово** в формате чекпоинта:
   ```
   ## Чекпоинт A: Backend поиска готов

   Что сделано:
   - ...
   - ...

   Как проверить (если хочешь сам):
   - `curl localhost:8000/notes/search?q=test`

   Дальше — UI поиска (шаги 5-7).

   Окей продолжать?
   ```
3. Жди ответ:
   - **ОК** → продолжай со следующего шага
   - **Стоп / правка** → обсуди с пользователем, при необходимости откатись или вернись в фазу 2

**Не показывай пользователю детали каждого пройденного шага — только итог чекпоинта.** Иначе диалог утонет в шуме.

### 5. Если в плане обнаружен пробел

В коде или контексте появилось что-то чего план не учёл:

**Не импровизируй.** Шансы что ты угадаешь намерение пользователя ниже чем шансы что ты добавишь не то.

Вместо этого:
1. Останови выполнение.
2. Опиши пробел пользователю простыми словами.
3. Предложи: *"Это пропустили в плане. Варианты: (а) я добавлю шаг прямо в план тут, (б) вернёмся в `/plan` и пересмотрим. Что выбираешь?"*

### 6. После всех шагов — проверь критерий готовности

Возьми критерий из задачи (фаза 1) и пройдись по нему:
- По каждому пункту — выполненная команда/действие И её реальный вывод. Пункт без наблюдаемого доказательства помечается ❌ (не ✅), отчёт не финализируется.
- Если проверить нечем — честно пиши "не проверено" и почему.
- Если что-то не выполнено → не финализируй, верни в работу

### 7. Финальный отчёт

```
## Задача выполнена

**Что сделано:** [1-2 предложения]

**Критерий готовности:**
- ✅/❌ [пункт 1]: [команда] → [вывод]
- ✅/❌ [пункт 2]: [команда] → [вывод]

**Файлы:** N изменено, M создано.

Открыть в редакторе? Запустить тесты целиком? Сделать коммит?
```

### Финал: GitHub-sync (тихо если выключен)

После того как критерий готовности пройден и пользователю показан финальный отчёт:

```bash
bash $CLAUDE_PLUGIN_ROOT/skills/github-sync/sync.sh close-task <task-slug>
bash $CLAUDE_PLUGIN_ROOT/skills/github-sync/sync.sh sync-all
```

- `close-task` — закрывает родительский Issue, ставит label `forge:done`, добавляет запись в `.forge/journal.yml`
- `sync-all` — обновляет шапку README.md и тело Pinned Issue (карта проекта)

Одной строкой в чат: "Карта проекта обновлена" (если sync включён).

### Триггер "перепривяжи" (перепривязать задачу к другой цели)

Если пользователь в любой момент во время выполнения говорит "перепривяжи задачу на цель X", "не туда привязал, перенеси на Y", "это к другой цели":

1. Получи открытые milestones: `gh api 'repos/{owner}/{repo}/milestones?state=open' --jq '.[] | "\(.number)|\(.title)"'`
2. Найди номер цели по имени (если пользователь сказал название) или спроси одной фразой какую цель имел в виду
3. `bash $CLAUDE_PLUGIN_ROOT/skills/github-sync/sync.sh reassign-task <task-slug> <new-milestone-num>`
4. Одной строкой: "Перепривязал на цель **'X'**"

## Что НЕ делаешь в этой фазе

- **Не сворачиваешь автоматически в коммит/push.** Спроси пользователя.
- **Не делаешь рефакторинг "пока я тут".** Меняешь только то что в плане.
- **Не пытаешься исправить смежные баги.** Это новая задача — иди в фазу 1 если нужно.
- **Не выполняешь шаги "сразу пачкой" игнорируя чекпоинты.** Чекпоинт = пауза, не формальность.
- **Не показываешь пользователю каждый прочитанный файл / каждый под-шаг субагента.** Только итоги шагов и чекпоинты.

## Пример хорошей и плохой делегации субагенту

❌ Плохо:
> Запустить субагента: "проверь весь код проекта"

Слишком общее — субагент потеряется, вернёт мусор.

✅ Хорошо:
> Запустить субагента: "Прочитай src/routes/*.py. Найди все эндпоинты которые не имеют декоратора `@requires_auth`. Верни список в формате `файл:строка функция_name` — больше ничего."

Конкретно что искать, конкретно что вернуть. Контекст основной сессии получит ровно нужное.

## Выход

Реализованная задача. Файлы изменены / созданы. Критерий готовности подтверждён.

Опционально (с подтверждения пользователя): коммит, обновление `.forge/journal.yml`, переход к следующей задаче.
