# Hookify — превратить повторное исправление в правило

Антон голосом часто повторяет одно и то же: "не используй emoji в коде", "не пиши комментарии 'TODO'", "коммиты только на русском в первой строке". Этот скилл фиксирует такие правила раз и навсегда — кладёт в `.forge/hookrules/*.md`, и PreToolUse-хук применяет их автоматически.

## Когда триггерится

**Явно:** Антон сказал "захукай это" / "сделай правилом" / "чтоб не повторялось" / "запомни и блокируй".

**Проактивно:** Claude замечает что Антон исправил его 3+ раз по одной теме в текущей сессии → предлагает: *"Ты уже 3 раза просил не Х. Зафиксировать как правило? (`/forge:hookify`)"*

## Процесс

### 1. Если есть аргумент (явное описание)

Пользователь сказал что-то типа "захукай чтобы не было tabs, только spaces". Идёшь сразу к шагу 3 — сформировать правило из описания.

### 2. Если аргумента нет (auto-режим)

Сделать **анализ последних 50 сообщений** текущей сессии:

- Где пользователь сказал "не", "не надо", "не так", "опять", "снова", "стоп", "нет, я же говорил", "блин", любую коррекцию?
- Сгруппировать коррекции по теме (что именно правил)
- Найти повторяющиеся (3+ раз) — это кандидаты на правило

Запустить субагент `conversation-analyzer` если есть. Иначе — сам пройди диалог.

Покажи Антону **топ-3 кандидата на правило**:
```
Я заметил 3 повторяющиеся коррекции:
1. Ты 4 раза просил не использовать emoji в коде → захукать?
2. Ты 3 раза правил коммиты на английский → на русский → захукать?
3. Ты 3 раза говорил не делать pip install без venv → захукать?

Какие из них правилом? (1/2/3/все/нет)
```

### 3. Сформировать правило

Спроси Антона **по одному вопросу за раз**:
1. **Что блокировать или предупреждать?** (regex или паттерн)
2. **Жёстко или мягко?** (block = команда не выполняется; warn = выполняется но с предупреждением)
3. **На каком инструменте срабатывает?** (Bash / Edit / Write / любой)
4. **Сообщение Claude когда правило сработало** (чтоб понимал почему)

### 4. Записать в `.forge/hookrules/<slug>.md`

Формат файла:

```markdown
---
name: no-emoji-in-code
matcher: Edit|Write
action: block
pattern: '[\\x{1F300}-\\x{1F9FF}]|[\\x{2600}-\\x{27BF}]'
message: "Антон просил не вставлять emoji в код. Если правда нужно — спроси явно."
created: 2026-05-14
---

## Контекст

В сессии 2026-05-14 Антон 4 раза просил убрать emoji из исходников.

## Примеры что блокировать
- emoji в комментариях кода
- emoji в strings
- emoji в filenames

## Когда правило НЕ применяется
- README.md (там emoji норм)
- HTML с эмодзи как фичей UI
```

### 5. Подтвердить и сохранить

Покажи Антону путь к файлу и кратко: *"Правило `no-emoji-in-code` создано. PreToolUse-хук подхватит его на следующем вызове Edit/Write — restart не нужен."*

### 6. (Опционально) дополнить bash-safety

Если правило про опасную команду (`rm`, `git push --force`, etc.) и оно более узкое чем существующие в `bash-safety.sh` — спросить: *"Добавить ещё в hardcoded `bash-safety.sh` чтобы работало даже без `.forge/hookrules/`?"*

## Чего НЕ делаешь

- **Не создаёшь правила без подтверждения.** Антон должен явно сказать "да, захукай".
- **Не дублируешь существующие правила.** Перед записью — прочитай `.forge/hookrules/*.md`, проверь нет ли уже такого.
- **Не пишешь правила которые блокируют 90% работы.** Спроси: "это специфический случай или общий?"
- **Не создаёшь правила из единственного случая.** Только если коррекция повторялась 3+ раз ИЛИ пользователь явно попросил.

## Выход

Файл `.forge/hookrules/<slug>.md` создан. PreToolUse-хук `hooks/user-rules-check.sh` (зарегистрирован в `hooks/hooks.json`) применит правило живьём при следующем вызове Bash/Edit/Write/NotebookEdit — restart не нужен.
