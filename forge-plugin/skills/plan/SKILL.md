# Plan — Phase 2: Building the route from A to B

Вход: чистая задача + критерий готовности из фазы 1 (`.forge/tasks/...`).
Выход: план с чекпоинтами в `.forge/plans/...`, готовый для фазы 3.

## Цель фазы

Превратить **что делать** (задача) в **что и в каком порядке** (план).

**Почему план важен:** реализация без плана = решения принимаются "на лету" под давлением кода. План = решения приняты холодно, на свежую голову, со временем подумать. Дешевле выбросить плохой план чем переписать плохой код.

План — это контракт, который фаза 3 (`/critique`) будет рвать на части. Чем конкретнее план — тем дешевле найти в нём дыры до того как начнём кодить.

## Что в плане ОБЯЗАНО быть

1. **Цель** — 1 предложение, ссылка на задачу из фазы 1.
2. **Подход** — 2-3 предложения как мы это делаем (общая идея, не детали).
3. **Список шагов** — каждый конкретный, тестируемый, маленький (2-15 минут работы).
4. **Чекпоинты** — между шагами, на смысловых границах.
5. **Открытые вопросы** — если что-то осталось неясным.

## Что в плане НЕ должно быть

- DDD-моделирование с bounded contexts / value objects / aggregates **по умолчанию**. Только если задача реально требует доменной сложности. Большинству фич это не нужно — это и есть тот формализм который превращает план в неприступную стену текста.
- Формальные requirements с R1/AC/Priority — критерий готовности уже в задаче из фазы 1, не дублируй.
- Список из 5 опций "execution mode" — это решается в фазе 3, не здесь.
- Технические вопросы пользователю — ищи сам.

## Главный принцип: разделение вопросов

- **Бизнес-логика → пользователю.** Что должно происходить, как видит конечный пользователь, что в каких случаях, что важнее.
- **Технические детали → сам.** Структура кода, API, библиотеки, что уже есть. Найди в коде / `.forge/library/*/spec.yml` / Context7.

**Почему так:** пользователь — не-кодер. Технические вопросы для него = шум который раздражает и не двигает задачу.

**Один вопрос за раз. Простой язык.**

## Process state (для statusline)

В начале фазы перезаписываем `.forge/state.yml`:

```bash
mkdir -p .forge && cat > .forge/state.yml <<EOF

```

## Процесс

### 1. Прочитай вход

`.forge/tasks/YYYY-MM-DD-<slug>.md` — задача + критерий из фазы 1.

Если файла нет — спроси: *"Делал ли `/new-task`? Без чистой задачи план будет на догадках."* Не пытайся восстановить задачу из истории чата — ты пропустишь нюансы.

### 2. Тихо изучи контекст

В этом порядке (не пиши пользователю что читаешь):

1. `.forge/library/*/spec.yml` — что уже сделано (читай это **до** кода)
2. L0 (`.forge/index.yml`) — стек, стадия
3. `.forge/dead-ends.yml` — что уже пробовали и не сработало (чтобы не повторить)
4. `.forge/decisions.yml` — архитектурные решения (чтобы не противоречить)
5. Конкретный код через Read/Grep если нужно
6. Документация библиотек через Context7 MCP
7. `WebSearch` если задача требует свежей информации

**Почему именно этот порядок:** spec.yml короче кода в 10 раз и содержит то что важно. Чтение кода без spec.yml = трата контекста.

### 3. Найди блокеры

После исследования спроси себя:
- Что мешает сделать задачу прямо сейчас?
- Есть ли крупное "до"?

**Два вида блокеров:**

#### Близкий блокер — становится шагом плана

Связан с задачей напрямую, маленький, без своих бизнес-вопросов.

**Пример:** задача "добавить поиск по заметкам". При исследовании выяснилось — в таблице `notes` нет колонки `text_search_vector` для индексации. → Это шаг плана: "Добавить колонку и миграцию".

#### Дальний блокер — выносим в отдельную сессию

Большой, отдельный, со своими бизнес-вопросами, загадит контекст основной сессии.

**Пример:** задача "добавить поиск по заметкам". При исследовании выяснилось — в проекте вообще нет аутентификации, а поиск должен быть только для залогиненного. → Это **отдельная задача с собственной фазой 1** ("какой провайдер? email/password или OAuth? нужны ли роли?..."). Решать её внутри плана поиска = смешать два разных дела.

**Что делаешь с дальним блокером:**

1. Скажи пользователю простыми словами:
   *"Чтобы сделать поиск нужна сначала авторизация — а её в проекте нет. Это большая отдельная история со своими вопросами (какой провайдер, нужны ли роли). Если решать прямо тут — забьём контекст и оба дела сделаем хуже. Предлагаю: ты открываешь новую сессию, я даю готовый промпт для неё, там проходим тот же пайплайн. Когда вернёшься — продолжим план поиска."*

2. Сформируй короткий самодостаточный промпт (не зависит от текущего чата):
   ```
   В проекте forge-plugin нужно добавить аутентификацию.
   Контекст: блокирует фичу поиска по заметкам.
   Текущий стек: см. .forge/index.yml.
   Дальше пройди /new-task → /plan → /critique → /execute.
   ```

3. Сохрани в `.forge/blockers/YYYY-MM-DD-<blocker-slug>.md` вместе с короткой запиской "что вернулось обратно даст этому плану".

4. Пометь в текущем плане: `**Блокер:** см. .forge/blockers/...`

5. Останови работу. Ждать пока пользователь не вернётся.

**Критерий "дальний vs близкий":** если решение блокера = новая задача с собственными бизнес-вопросами → дальний. Если "создать недостающий файл / поле / индекс" → близкий.

### 4. Задай вопросы по бизнес-логике (если остались)

Только то на что не отвечает ни задача, ни код. Один вопрос за раз.

### 5. Спроектируй план

#### Бит-сайз шаги

Каждый шаг = одно действие 2-15 минут.

**Хороший шаг:**
```
## Шаг 3: Добавить эндпоинт GET /notes/search

**Файлы:**
- Создать: `src/routes/notes_search.py`
- Изменить: `src/app.py` — зарегистрировать роут

**Что делаем:**
```python
@router.get("/notes/search")
async def search_notes(q: str, user_id: int = Depends(get_current_user)):
    return await note_repo.search(q, user_id)
```

**Как проверим:** `curl localhost:8000/notes/search?q=test` возвращает JSON-список совпадений для текущего пользователя.
```

**Плохой шаг (так не делай):**
```
## Шаг 3: Сделать backend для поиска

Добавить валидацию, обработку ошибок, оптимизацию.
```

Почему плохо: непонятно что делать, нет файлов, нет проверки. Это не шаг — это половина задачи.

#### TDD — где даёт смысл

Используй для функциональной логики: парсеры, валидаторы, бизнес-правила, расчёты, утилиты.

Не насаждай TDD на: HTML-вёрстку, конфиги, миграции БД, регистрацию роутов, импорты.

**Почему:** TDD на конфиге = тратить время на тест который говорит "конфиг прочитался". Бесполезно.

#### Точные пути файлов

`src/models/note.py`, не "файл с моделью".

#### Концентрированный код

Если шаг про код — пиши код в шаге, не "добавить валидацию".

### 6. Расставь чекпоинты

**Чекпоинт = пауза где пользователь смотрит результат и говорит ОК.**

#### Где ставить чекпоинт

- Завершена целостная фича или её слой (БД готова → переход к API)
- Перед миграцией данных или необратимой операцией
- Перед массовым изменением (рефакторинг 10+ файлов)
- Перед добавлением зависимости / внешнего сервиса
- На границе ответственности (backend закончили — переходим к UI)

#### Где НЕ ставить

- Через каждые N шагов "для безопасности"
- На каждом коммите (коммиты и так есть в TDD-шагах)
- На каждом мелком шаге — пользователь устанет подтверждать

**Почему чекпоинты не каждые N шагов:** искусственная пауза прерывает поток, не даёт ничего проверить (рано), и приучает пользователя проштамповывать "ОК" не глядя. Чекпоинт работает только когда есть что показать.

**Формат чекпоинта в плане:**
```markdown
---

### ✅ Чекпоинт A: Backend поиска готов

Что показываем: `curl localhost:8000/notes/search?q=...` возвращает JSON.
Что подтверждает пользователь: "поиск работает на уровне API".

Следующее: UI поиска.

---
```

### 7. Запиши план

В `.forge/plans/YYYY-MM-DD-<slug>.md`:

```markdown
# План: [название задачи]

**Задача:** см. `.forge/tasks/YYYY-MM-DD-<slug>.md`

**Подход:** [2-3 предложения]

**Открытые вопросы:** [список или "нет"]

**Блокеры:** [список ссылок на `.forge/blockers/...` или "нет"]

---

## Шаг 1: ...
## Шаг 2: ...

### ✅ Чекпоинт A: ...

## Шаг 3: ...
```

### 7.5. GitHub-sync (опционально, тихо если выключен)

```bash
bash $CLAUDE_PLUGIN_ROOT/skills/github-sync/sync.sh add-steps <task-slug> .forge/plans/YYYY-MM-DD-<slug>.md
```

Создаёт sub-issues под родительским Issue (если он был создан в Phase 1). Идемпотентно — повторный вызов ничего не дублирует. Если sync выключен или нет parent Issue — молча no-op.

### 8. Покажи план пользователю

Кратко (не дамп всего файла):
*"План готов: N шагов, M чекпоинтов. Лежит в `.forge/plans/...`. Открой посмотри. Если ОК — запускаю критику. Скажи 'стоп' если хочешь паузу."*

Покажи сводку плана и **заверши ход — дождись ответа**. Если пользователь ответил 'ОК' / 'да' / 'погнали' — инвокни critique skill (Phase 3). Если 'стоп' / правки — примени правки и снова покажи сводку.

## Что НЕ делаешь в этой фазе

- **Не пишешь production-код.** Это фаза 4.
- **Не предлагаешь "режимы выполнения".** Это фаза 3 (там критики дописывают как выполнять).
- **Не формализуешь requirements** — критерий уже в задаче.
- **Не задаёшь технических вопросов пользователю.**

## Выход

Файл `.forge/plans/YYYY-MM-DD-<slug>.md`.

Передаётся в фазу 3 — `/critique`.
