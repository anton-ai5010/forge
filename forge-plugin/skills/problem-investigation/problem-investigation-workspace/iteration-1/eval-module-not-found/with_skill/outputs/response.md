Это классическая ситуация, когда симптом (модуль не найден) и реальная причина (почему Node его не видит) — разные вещи. Файл может быть на месте, но Node резолвит путь не оттуда, откуда ты думаешь.

Прежде чем копать в коде, один вопрос: **это работало раньше, или это первый запуск после какого-то изменения?** (npm install, переименование файлов, переезд в другую директорию, обновление зависимостей)

Пока жду ответ, вот что я бы проверил (и проверю, как только подтвердишь контекст):

**Наиболее частые причины именно этой ошибки, когда файл "точно есть":**

1. **Расширение файла.** Импорт `'../lib/utils'` — а файл называется `utils.ts`, а в проекте не настроен резолвинг `.ts` расширений (например, нет ts-node, или в webpack/vite не прописаны resolve.extensions).

2. **Регистр в пути.** На Linux файловая система case-sensitive. Если файл называется `Utils.js`, а импорт `'../lib/utils'` — Node его не найдет. На macOS это бы работало, на Linux — нет.

3. **Импорт идет не оттуда, откуда кажется.** Относительный путь `../lib/utils` резолвится от файла, который делает импорт. Если после рефакторинга файл-импортер переехал на уровень глубже — путь сломался, хотя `lib/utils` на месте.

4. **Скомпилированные файлы.** Если `npm run dev` запускает что-то из `dist/` или `build/`, там может лежать старая скомпилированная версия с устаревшим путем, а ты смотришь на исходники в `src/`.

5. **Симлинки/node_modules.** Если `lib/utils` — это пакет из монорепы, симлинк мог сломаться после `npm install`.
