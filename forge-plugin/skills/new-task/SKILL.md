# New Task — Phase 1: Understanding

Превращает сырой промпт пользователя в **чистую задачу + критерий готовности**.

Без этой фазы реализация строится на догадках: половина решений делается "по умолчанию" и пользователь обнаруживает их только когда смотрит результат. Дешевле прояснить сейчас одним вопросом, чем переделывать потом.

## Главный принцип: разделение вопросов

Перед каждым вопросом спроси себя: **"могу ли я найти этот ответ сам?"**

- **Логические → пользователю.** Только он знает: что должно происходить, кому адресовано, что важнее (скорость / точность / UX), как должно вести себя в краевых случаях с его точки зрения, какой смысл имеет фича для его дела.
- **Технические → ищешь сам.** Какой стек, где лежит код, какие библиотеки, как устроена текущая архитектура, какой API, какие колонки в таблице — читаешь код / `.forge/library/*/spec.yml` / доки / Context7. Пользователя не дёргаешь.

**Почему это важно:** пользователь forge-plugin часто не-кодер. Технические вопросы для него = ощущение что его заваливают непонятным. Логические — только он на них может ответить, и без этих ответов задача поставлена криво. Если задал техникум — это твоя ошибка, найди сам.

### Пример разделения

Пользователь говорит: *"хочу добавить в проект поиск"*

**Спросить пользователя (логические):**
- Что искать — заметки / задачи / пользователей / весь контент?
- Кому это нужно — только админу или всем залогиненным?
- Что важнее — скорость ответа или точность совпадений?
- Нужна история поиска / автодополнение / фильтры по дате?

**Найти самому (технические):**
- Стек проекта → `package.json` / `pyproject.toml` / `.forge/index.yml`
- База данных и какие таблицы есть → `.forge/library/*/spec.yml` или код моделей
- Что из поиска уже реализовано → grep по проекту
- Подходящие библиотеки под стек → Context7 MCP

## Правила общения

- **Один вопрос за раз.** Никаких списков "ответь на эти 5 вопросов". Один — ответ — следующий. Длинные списки парализуют, особенно не-кодера.
- **Простой язык.** Объясняешь как первокурснику. Без жаргона. Коротко. Если можешь сказать в 2 строки — скажи в 2.
- **Мультиагент внутри, один голос снаружи.** Перед каждым вопросом внутренне прокрути: "Скептик: чего я не учёл? Адвокат пользователя: что я пропустил?" Наружу — один чёткий вопрос с учётом этих углов. Не показывай пользователю что внутри 4 голоса — он видит одного помощника.

## Process state (для statusline)

В самом начале фазы запиши состояние пайплайна в `.forge/state.yml` через Bash:

```bash
mkdir -p .forge && cat > .forge/state.yml <<EOF
phase: new-task
task: <короткий-slug-из-промпта>
started_at: $(date -Iseconds)
EOF
```

Когда фаза завершится и Claude перейдёт в `/plan` — новый скилл перезапишет state.yml сам.

## Процесс

1. **Прочитай сырой промпт.** Не реагируй сразу. Поставь паузу — что тут точно ясно, а что предположение?

2. **Тихо изучи технический контекст.** В этом порядке:
   - `.forge/library/*/spec.yml` если есть — там сжатые описания того что уже сделано
   - L0 (`.forge/index.yml`) — стек, стадия проекта, текущая задача
   - При необходимости — конкретный код через Read/Grep
   - Библиотеки/фреймворки через Context7 MCP если упомянуты
   - **Не пиши пользователю что ты сейчас исследуешь.** Это твоя кухня.

3. **Внутренне составь список логических неясностей.** Каждый пункт — то на что **только пользователь** может ответить.

4. **Задай один вопрос.** Самый блокирующий — без ответа на который остальное не имеет смысла. Подожди ответ.

5. **Повторяй шаг 4** пока всё не ясно. Между вопросами — снова мультиагент-проверка.

6. **Если пользователь буксует** (говорит "не знаю", "не могу объяснить", "не получается сформулировать") — предложи HTML-набросок. См. ниже.

7. **Когда понял всё** — переформулируй в формате:
   ```markdown
   ## Задача
   [1-2 предложения чем мы занимаемся]

   ## Критерий готовности
   - [тестируемый пункт 1]
   - [тестируемый пункт 2]
   - [тестируемый пункт 3]
   ```

   **Критерий должен быть наблюдаемым.** Не "работает хорошо", а "когда пользователь делает X, на экране Y". Если ты сам не можешь проверить выполнение — критерий плохой.

8. **Покажи пользователю. Спроси одним вопросом:** "Да, оно? Или поправить?"

9. **Если ОК** — сохрани в `.forge/tasks/YYYY-MM-DD-<slug>.md` и сообщи:
   *"Задача зафиксирована. Перехожу к плану. Скажи 'стоп' если хочешь паузу."*

   Затем **сразу инвокни plan skill** (Phase 2). Не жди явного `/plan` от пользователя — он уже подтвердил, цепочка идёт автоматически. Останься в фазе 1 только если пользователь сказал 'стоп' / 'пауза' / 'погоди' / любой эквивалент.

10. **Если нет** — назад к вопросам, начиная с того что не так.

### Пример хорошего выхода

```markdown
## Задача
Добавить страницу поиска по заметкам пользователя — ищет по заголовку и тексту, доступна только залогиненному.

## Критерий готовности
- На /search есть поле ввода; при вводе 3+ символов появляется список совпадений в реальном времени
- В выдаче — заголовок, дата, первые 100 символов текста с подсвеченным совпадением
- При клике на результат — открывается полная заметка
- Незалогиненного редиректит на /login
```

### Пример плохого выхода (так НЕ делай)

```markdown
## Задача
Сделать поиск.

## Критерий готовности
- Работает хорошо
- Быстрый
- Удобный
```

Это бесполезно — невозможно проверить. Каждый пункт должен быть конкретным наблюдаемым событием.

## HTML-набросок (опционально)

Запускай когда:
- Пользователь явно попросил "покажи картинкой" / "нарисуй"
- Пользователь буксует: "не знаю как объяснить", "не могу сформулировать"
- Задача про UI / поток / структуру данных — там где картинка эффективнее слов

Что делаешь:
- Рисуешь **одно-экранный HTML** — карточки, простые блоки, без интерактива
- Сохраняешь в `.forge/sketches/<task-slug>.html`
- Даёшь `file://` ссылку
- Используешь набросок как новую точку диалога: *"Что здесь не так? Чего не хватает?"*

**Картинка — не финальный дизайн.** Это зацепка для мысли пользователя. Кривой набросок который пользователь поправит лучше "правильного" наброска который пользователь не может объяснить.

## Чего НЕ делаешь в этой фазе

- **Не предлагаешь подходы / архитектуру.** Это фаза 2 (`/plan`).
- **Не пишешь формальные requirements с R1/AC/Priority.** Только "Задача + Критерий". Формализм здесь — overkill.
- **Не делаешь домен-моделирование, bounded contexts, value objects.** Если вообще понадобится — это фаза 2.
- **Не запускаешь 3 агента для интернет-ресёрча конкурентов.** Это фаза 2.
- **Не задаёшь технических вопросов** ("какой фреймворк?", "Postgres или MySQL?"). Найди сам или оставь решение на фазу 2.
- **Не дампишь 5 вопросов одним сообщением.** Один.

## Выход

Файл `.forge/tasks/YYYY-MM-DD-<slug>.md` с двумя секциями: `## Задача` и `## Критерий готовности`.

Передаётся в фазу 2 — `/plan`.
