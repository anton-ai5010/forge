# Critique — Phase 3: Tearing the plan apart cheaply

Вход: план из фазы 2 (`.forge/plans/...`).
Выход: финальный план с правками + секция "как выполнять", готовый для фазы 4.

## Цель фазы

Найти дыры в плане **до** того как код написан. Дыра в плане стоит правки одного абзаца. Та же дыра в коде = переделать 10 файлов и потерять день.

Дополнительно: дописать **как выполнять** план — что в субагентов, что параллельно, что последовательно. Это решение тоже принимается холодной головой, а не "по ходу" в фазе 4.

**Почему критика — отдельная фаза, а не часть фазы 2:**
- Автор плана влюблён в свой план. Просить его же критиковать = самообман.
- Свежий агент со специфической ролью находит реальные проблемы, а не ставит галочки.
- Параллельные персоны бесплатны по времени (запускаются одновременно).

## Process state (для statusline)

В начале фазы перезаписываем `.forge/state.yml`:

```bash
mkdir -p .forge && cat > .forge/state.yml <<EOF
phase: critique
task: <slug-из-плана>
started_at: $(date -Iseconds)
EOF
```

## Процесс

### 1. Прочитай план и задачу

- `.forge/plans/YYYY-MM-DD-<slug>.md` — план из фазы 2.
- `.forge/tasks/YYYY-MM-DD-<slug>.md` — задача из фазы 1 (чтобы критики знали критерий готовности).
- Если плана нет — спроси: *"Делал ли `/plan`? Без плана нечего критиковать."*

### 2. Запусти 4 персоны параллельно

**В одном сообщении** запусти 4 субагента через Agent tool. Не последовательно — параллельно. Каждой персоне дай:
- Полный текст плана
- Полный текст задачи (для контекста критерия)
- Её специфический prompt (см. ниже)

#### Персона 1: Скептик (что сломается?)

```
Ты — Скептик. Твоя работа: найти всё что может пойти не так в этом плане.

Думай про:
- Краевые случаи (пустой ввод, очень большой ввод, null, спецсимволы, конкурентные операции)
- Что если внешний сервис недоступен, тормозит, возвращает ошибку
- Что если данных в БД нет / уже есть / больше чем ожидаем
- Что если пользователь сделает не то что мы ожидаем (двойной клик, бэк, F5)
- Что в плане осталось "по умолчанию" — то есть Claude решит сам, а решит, скорее всего, неправильно
- Что может сломать существующие фичи (регрессии)

Верни список конкретных рисков. По каждому риску: ЧТО сломается, ГДЕ в плане слабое место, КАК поправить (конкретная правка плана).

Не пиши общих фраз "нужно подумать про обработку ошибок". Пиши конкретно: "Шаг 3 не учитывает пустой результат поиска — добавить отдельный шаг 3.5: вернуть {results: [], message: 'ничего не найдено'}".

Формат КАЖДОЙ находки:
- Риск: ...
  Где в плане: Шаг N
  Правка: ...
  Confidence: 0-100 (насколько ты уверен что это реальная проблема, не "может быть")

**Confidence критерий:**
- 90-100: явная регрессия / потеря данных / стопроцентный failure case
- 70-89: высоковероятная проблема под нагрузкой / в edge case
- 50-69: возможная проблема — но зависит от контекста
- <50: догадка / стилистика — НЕ пиши такие, отфильтруй сам

Финальный список — только находки с confidence ≥ 80%. Если ниже — отбрось.

Если ничего ≥ 80% не нашёл — пиши "Серьёзных рисков с confidence ≥ 80% не обнаружено" и поясни почему план достаточно надёжен.
```

#### Персона 2: Прагматик (можно проще?)

```
Ты — Прагматик. Твоя работа: найти где в плане лишняя сложность.

Думай про:
- Можно ли решить задачу меньшим числом шагов?
- Есть ли в плане абстракции / слои / паттерны которые не нужны (one-time use, premature)?
- Есть ли в плане фичи которых пользователь не просил?
- Можно ли использовать существующий код / библиотеку вместо своего?
- Есть ли шаги которые делают то же самое разными способами?
- Нет ли в плане "improvements" к коду который не часть задачи?

Принцип: 3 одинаковые строки лучше преждевременной абстракции. YAGNI безжалостно.

Исключение: HTML-макет перед UI-кодом и секция «Что увидишь ты» — правила пайплайна forge, а не лишние фичи. Не предлагай их убирать.

Формат КАЖДОЙ находки:
- Лишнее: ...
  Где в плане: Шаг N
  Упрощение: ...
  Confidence: 0-100 (насколько уверен что это перебор)

Возвращай только находки с confidence ≥ 80%. Ниже — отбрось.

Если всё минимально и confidence ≥80% не нашёл — пиши "План минимален" и поясни.
```

#### Персона 3: Архитектор (масштабируется? переиспользуется? противоречит ли существующему?)

```
Ты — Архитектор. Твоя работа: проверить план на согласованность с проектом.

Сначала прочитай:
- .forge/library/*/spec.yml (что уже есть)
- .forge/decisions.yml (какие архитектурные решения приняты)
- .forge/conventions.yml (правила naming / структуры)
- .forge/map.yml (red zones — куда нельзя трогать)

Потом думай про:
- Не противоречит ли план существующим решениям?
- Не повторяет ли план уже сделанное?
- Не нарушает ли naming/structure conventions?
- Не трогает ли план red zones из map.yml?
- Если задача масштабная — выдержит ли план рост (10x пользователей, 100x данных)?
- Не создаёт ли план циклических зависимостей между модулями?

Не выдумывай теоретические сценарии масштаба ("а вдруг 1 миллион запросов в секунду"). Думай только про реальные ограничения проекта.

Формат КАЖДОЙ находки:
- Несогласованность: ...
  Где в плане: Шаг N
  Правка: ...
  Confidence: 0-100 (насколько уверен что это реальное противоречие)

Возвращай только с confidence ≥ 80%. Ниже — отбрось.

Если всё согласовано (и нет находок ≥80%) — пиши "План согласован с архитектурой" и подтверди какие точки проверил.
```

#### Персона 4: Адвокат пользователя (поймёт ли человек?)

```
Ты — Адвокат пользователя. Пользователь — не-кодер. Твоя работа: проверить план с его стороны.

Думай про:
- Будет ли понятно конечному пользователю что происходит на экране?
- Получит ли он понятную ошибку если что-то пойдёт не так?
- Не требуется ли от него технических знаний (типа "введите ID")?
- Есть ли индикация прогресса для долгих операций?
- Что увидит пользователь когда нет данных / только начал?
- Соответствует ли план критерию готовности из задачи — а критерий покрывает реальные сценарии использования?

Не критикуй UI который не часть задачи. Критикуй только то что план прописывает или подразумевает.

Формат КАЖДОЙ находки:
- UX-проблема: ...
  Где в плане: Шаг N
  Правка: ...
  Confidence: 0-100 (насколько уверен что это реально не-кодер не поймёт/не справится)

Возвращай только с confidence ≥ 80%. Ниже — отбрось.

Если UX продуман (нет находок ≥80%) — пиши "С точки зрения пользователя всё ок" и почему.
```

### 3. Дождись все 4 отчёта

Параллельно — значит запустить в одном сообщении 4 вызова Agent tool. Подожди все ответы.

### 4. Синтезируй

После получения всех отчётов:

1. **Фильтр confidence ≥ 80%.** Каждая персона помечает свои находки уверенностью (0-100). Игнорируй всё что <80%. Это режет шум (стилистические придирки, паранойя без основания) и оставляет только реальные проблемы. Если в отчёте персоны нет confidence — попроси переоценить (или применяй default 50% = отсеять).

2. Объедини оставшиеся правки в один список. Убери дубли (если 2 персоны указывают на тот же шаг с похожей сутью — мерджи их, итоговый confidence = max).

3. Каждой правке поставь приоритет:
   - **Блокер** — без этого план нельзя выполнять (риски: данные потеряются, регрессия, нарушение red zones)
   - **Важно** — план будет работать, но плохо
   - **Опционально** — мелочи / "было бы хорошо"

4. Покажи пользователю кратко (не дамп всех отчётов):
```
## Критика плана

**Блокеры (N):**
- ...

**Важное (M):**
- ...

**Опционально (K):**
- ...

Применить блокеры + важное? (опциональные оставим если есть время)
```

5. Подожди ответ. После ОК — обновляй план в `.forge/plans/...` с правками.

### 4.5. GitHub-sync (опционально, тихо если выключен)

Запиши сводку блокеров + важного в `/tmp/forge-critique-<task-slug>.md` (формат markdown с заголовками "## Блокеры" и "## Важное"), затем:

```bash
bash $CLAUDE_PLUGIN_ROOT/skills/github-sync/sync.sh add-critique <task-slug> /tmp/forge-critique-<task-slug>.md
```

Добавит комментарий в родительский Issue + сменит label на `forge:phase-3`. Молча no-op если sync выключен.

### 5. Допиши секцию "Как выполнять"

После применения правок, добавь в конец плана секцию **Execution Strategy**:

```markdown
## Execution Strategy

### Параллельно (одновременно через субагентов)
- Шаги X, Y, Z — независимы друг от друга, делать одновременно

### Последовательно
- Шаги A → B → C — есть зависимости

### Делегировать субагентам (грязная работа)
- Шаг N: прочитать 50 файлов и составить список — в субагент (вернёт короткий итог)
- Шаг M: запустить тесты и проанализировать падения — в субагент

### Делать в основной сессии
- Шаги связанные с твоими решениями
- Финальная стыковка

### Чекпоинты
- После шага N — пауза, показ результата
- После шага M — пауза, показ результата
```

**Принципы решения "что куда":**
- Тяжёлое чтение/запись/анализ файлов = субагент (изолированный контекст)
- Решения / стыковка / диалог с пользователем = основная сессия
- Независимые шаги = параллельно
- Зависимые / необратимые / критичные = последовательно

### 6. Покажи пользователю готовый план + реши про handoff

#### Сначала оцени тяжесть контекста сессии

Перед тем как выбрать как перейти к фазе 4, ответь себе **честно**:
- Сколько turns прошло в этом чате? (если 30+ — контекст уже забит)
- Сколько файлов прочитал в этой сессии? (10+ — много)
- План большой? (10+ шагов или будет много кода — да)
- Использовал ли /critique с 4 персонами (большие промпты субагентов остались в логах)? — да всегда здесь

**Это рекомендация Anthropic:** для тяжёлого `/execute` лучше свежая сессия — контекст планирования и критики не нужен в фазе реализации.

#### Если контекст УЖЕ тяжёлый (любое 2+ из условий выше):

Скажи пользователю:
*"План готов в `.forge/plans/<file>.md`. Применено N правок, Execution Strategy дописана.*

*Контекст этого чата уже большой — для качественной реализации рекомендую открыть новый чат и сказать: `/forge:execute план .forge/plans/<file>.md`. Свежий контекст = меньше ошибок к концу реализации.*

*Если хочешь продолжить здесь — скажи 'продолжай тут', и я запущу /execute прямо сейчас."*

**Не инвокай execute skill автоматически.** Жди явного решения пользователя.

#### Если контекст ЛЁГКИЙ (мало turns, мало файлов, простой план):

Скажи: *"План доработан: N правок, Execution Strategy дописана. Лежит в `.forge/plans/<file>.md`. Перехожу к реализации. Скажи 'стоп' если пауза, 'в новый чат' если хочешь чистый контекст."*

Затем **сразу инвокни execute skill** (Phase 4) если пользователь подтвердил. Останься в фазе 3 только если он сказал 'стоп' / 'пауза' / 'в новый чат'.

## Что НЕ делаешь в этой фазе

- **Не пишешь production-код.** Это фаза 4.
- **Не делаешь новый план с нуля.** Если план настолько плох что нужен пересмотр — скажи это пользователю и предложи вернуться в фазу 2.
- **Не критикуешь задачу из фазы 1** — её время прошло, мы критикуем план.
- **Не запускаешь персон последовательно** — параллельно или критика теряет смысл (одна влияет на другую).
- **Не показываешь пользователю полные отчёты 4 персон.** Только синтез.

## Пример хорошего отчёта Скептика

❌ Плохо:
> Нужно добавить обработку ошибок в шаге 3.

✅ Хорошо:
> Риск: Шаг 3 (поиск по заметкам) не обрабатывает случай когда `q` пустой или короче 3 символов — упадёт запрос в БД с непредсказуемым LIKE.
> Где в плане: Шаг 3
> Правка: Добавить в начало handler-а: `if not q or len(q) < 3: return {results: [], hint: "Введите минимум 3 символа"}`

## Выход

Обновлённый файл `.forge/plans/YYYY-MM-DD-<slug>.md` с применёнными правками и секцией Execution Strategy.

Передаётся в фазу 4 — `/execute`.
