# Forge Plugin — Evals

> **Статус: каркас.** Корпус не собран, ни один прогон не запускался. Этапы ниже — план работ, а не отчёт о сделанном.

Эвалы плагина по методологии Hamel Husain. Не синтетика, а трейсы из реальных сессий Антона (будут собраны на Этапе 1).

## Зачем

`tests/skill-triggering/` проверяет одну вещь: триггерится ли скилл по одному промпту. Это полезно, но не покрывает где плагин ломается на практике — переходы между фазами, голосовой ввод с опечатками, неправильный выбор скилла, etc.

Этот эвал-сетап будет показывать **карту реальных провалов** — планируется на базе ~116 сессий Антона (пока не собрано ни одной).

## Воркфлоу

```
1. Сбор корпуса        → corpus/raw/*.jsonl
2. Error analysis      → error-analysis.tsv (open coding)
3. Таксономия          → taxonomy-v1.md (axial coding)
4. Бинарные критерии   → criteria/*.yml
5. Прогон + анализ     → runs/YYYY-MM-DD/
6. Transition matrix   → transition-matrix.tsv
```

## Этап 1: Сбор корпуса (30 минут)

```bash
# Найти все JSONL трейсы Антона
find ~/.claude/projects -name "*.jsonl" -mtime -180 | head -50

# Скопировать в corpus/raw/
# Имена: YYYY-MM-DD-<short-topic>.jsonl
# Фильтр: оставлять только многоходовые (3+ user turns)
```

Цель: **30 трейсов** для старта. Не редактировать — голосовые опечатки = часть данных.

## Этап 2: Error analysis (60-90 минут)

**Open coding:** открой по одному трейсу в редакторе. Запиши **первый сбой** (downstream ошибки = последствия первого) в `error-analysis.tsv` свободным текстом. Не категоризируй пока.

Шаблон строки в `error-analysis.tsv`:
```
trace_id	phase	first_failure	open_code	axial_category	pass_fail	notes
2026-05-13-search	plan	Не понял про поиск по тексту или по тегам	question-too-vague		fail	
```

**Axial coding (после 20+ трейсов):** скопируй все `open_code` в один промпт Claude:
> "Сгруппируй эти 20 сбоев в 5-8 категорий. Дай частоты. Найди топ-3 по Парето."

Результат → `taxonomy-v1.md`. Проверь руками.

## Этап 3: Бинарные критерии

По одному `criteria/*.yml` на каждую фазу + handoff. Шаблон:

```yaml
phase: plan
checks:
  - id: plan_has_testable_goals
    question: "План содержит измеримые критерии успеха?"
    type: binary
  - id: plan_addresses_user_intent
    question: "План решает то что Антон попросил, не соседнюю задачу?"
    type: binary
```

**Binary только.** Likert (1-5) запрещён — субъективно.

## Этап 4: Прогон

```bash
bash run-evals.sh
# Прогоняет corpus/raw/* через текущий плагин
# Сохраняет в runs/$(date +%F)/
# Антон руками проставляет pass/fail (~15 мин на 20 трейсов)
```

## Этап 5: Transition Failure Matrix

После прогона обновить `transition-matrix.tsv`:

```
              | unblocker | new-task | refine-idea | plan | critique | execute | END
unblocker     |     -     |    2     |      0      |  0   |    0     |    0    |  0
new-task      |     0     |    -     |      3      |  1   |    0     |    0    |  0
refine-idea   |     0     |    0     |      -      |  2   |    0     |    0    |  0
plan          |     0     |    0     |      0      |  -   |    8     |    2    |  1
critique      |     0     |    0     |      0      |  4   |    -     |    6    |  0
execute       |     0     |    0     |      0      |  1   |    0     |    -    |  9
```

(числа в примере иллюстративные)

Строки = последнее успешное состояние. Столбцы = где первый сбой. **Hot cells** (плотные клетки) = приоритет дебага.

## Что НЕ делать

- Не покупать LLM-as-judge сразу — сначала 30+ трейсов глазами
- Не использовать generic метрики (ROUGE, BERTScore) — обманчиво
- Не писать тесты до error analysis — это eval-driven development, антипаттерн
- Не растягивать на Likert — бинарка
- Не строить инфраструктуру первой — TSV хватит

## Когда прогонять

- После каждого изменения скилла или хука (можно git pre-push)
- Перед каждым `Bump to vX.Y.Z` коммитом (gate релиза)
- Не нужно для правки docs/README

## Метрики

- **Pass rate:** % трейсов прошедших все критерии. **70% — норм.** 100% = слишком лёгкие тесты.
- **Регрессия:** категория где было 5/10, стало 2/10 → откатить или фиксить.
