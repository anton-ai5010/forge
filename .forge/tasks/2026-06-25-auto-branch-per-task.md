## Задача

Встроить в forge модель «одна задача = одна ветка → мерж», чтобы пользователь-не-кодер не держал в голове git вручную. На входе в `/forge:execute` (когда реально начинается код) forge сам заводит git-ветку `feat/<slug>`, а скилл `finishing-a-development-branch` в конце коммитит несохранённое, мержит и чистит ветку. Worktree и ручные коммиты по ходу из ментальной модели пользователя убираются.

## Критерий готовности

- При запуске `/forge:execute`, когда пользователь на основной ветке (master/main), ветка `feat/<slug>` создаётся молча (без вопросов), имя slug совпадает с именем файла плана `.forge/plans/YYYY-MM-DD-<slug>.md`. Незакоммиченные правки на основной ветке забираются на новую ветку автоматически.
- Если на входе в execute пользователь НЕ на основной ветке (сидит в другой feat-ветке) — forge не создаёт ветку молча: сообщает «ты на ветке X, она не слита в master» и одним вопросом предлагает начать от master (дефолт) или продолжить поверх текущей.
- Если пользователь уже на ветке `feat/<slug>` этой задачи — forge ничего не делает.
- forge НЕ угадывает связанность задачи с веткой молча — при «не на основной ветке» всегда спрашивает.
- При команде «мержим» / «всё готово» скилл `finishing-a-development-branch` коммитит несохранённую работу перед мержем, мержит ветку в master и удаляет её — пользователю не нужно отдельно коммитить по ходу.
- Документация (CLAUDE.md) отражает новую модель: ветка рождается в execute, worktree помечен как опциональный для параллельных задач, ручные коммиты по ходу больше не обязательны.
