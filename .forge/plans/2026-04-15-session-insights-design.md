# Session Insights — Design Doc

## Requirements

### Must-Have

**R1:** Скилл находит JSONL файлы текущего проекта
AC: Определяет project slug из cwd, читает все `~/.claude/projects/{slug}/*.jsonl`

**R2:** Извлечение только текстовых сообщений пользователя через bash/python
AC: Фильтрует `type=="user"`, достаёт `message.content[].text`, отбрасывает tool_result, IDE-контекст, interrupted-маркеры

**R3:** Дедупликация сессий
AC: Одинаковые сессии (дублирующиеся JSONL) не анализируются дважды

**R4:** Мультиагентный анализ — 4 параллельных агента после извлечения
AC: Агент извлечения → Агенты A-D параллельно (боли, темы, нерешённое, профиль)

**R5:** Категории инсайтов — все 6
AC: Отчёт содержит: болевые точки, повторяющиеся темы, нерешённые проблемы, эволюция идей, стиль общения, профиль пользователя

**R6:** Текстовый отчёт в консоли (markdown)
AC: Markdown-отчёт выводится пользователю в конце работы

**R7:** Автосохранение инсайтов в memory
AC: Создаёт/обновляет файлы в `~/.claude/projects/{slug}/memory/` с frontmatter (name, description, type) и обновляет MEMORY.md

**R8:** Защита от переполнения контекста
AC: Не загружает сырые JSONL в контекст агентов. Файлы >5МБ — только последние 500 user-сообщений

### Nice-to-Have

**R9:** Взвешивание по свежести — недавние сессии важнее
**R10:** Не дублировать существующие memory — обновлять

## Research Findings

- Аналогов для качественного анализа (боли, эволюция идей) нет — только количественные (claude-session-analyzer, CodeBurn)
- JSONL файлы бывают 3-12 МБ, tool_result раздувает размер — обязательна предфильтрация
- Русский текст лучше анализировать через Claude (не NLP библиотеки) — понимает контекст, иронию
- Гибрид bash-извлечение + LLM-анализ — оптимальный подход

## Architecture

### 3 фазы

**Фаза 1 — Извлечение (основной процесс, bash/python)**
1. Определить project slug: `pwd | sed 's|/|-|g' | sed 's|^-||'`
2. Найти файлы: `~/.claude/projects/{slug}/*.jsonl`
3. Python-скрипт извлекает user text messages с timestamps и session ID
4. Дедупликация по первым 3 сообщениям сессии
5. Сортировка по timestamp
6. Сохранение в `/tmp/session-insights-{slug}.txt`
   Формат: `[SESSION:{id8}] [TIMESTAMP] message_text`

**Фаза 2 — Анализ (4 параллельных агента)**

Каждый агент читает `/tmp/session-insights-{slug}.txt`:

- **Агент A — Боли и фрустрации:** негатив, прерывания, повторные просьбы, маркеры "не работает", "не понимаю", "стоп", "погоди"
- **Агент B — Темы и эволюция:** кластеризация по темам, повторяющиеся запросы, как менялись идеи между сессиями
- **Агент C — Нерешённое:** обсуждения без завершения, вопросы без ответов, брошенные ветки
- **Агент D — Профиль и стиль:** предпочтения, роль, экспертиза, голосовой vs текстовый ввод, паттерны общения

**Фаза 3 — Синтез (основной процесс)**
1. Собрать результаты 4 агентов
2. Сформировать markdown-отчёт (6 секций)
3. Вывести пользователю
4. Записать/обновить memory-файлы с frontmatter
5. Обновить MEMORY.md

## Data Flow

```
$PWD → project slug → ~/.claude/projects/{slug}/*.jsonl
  → python extraction → /tmp/session-insights-{slug}.txt
    → 4 parallel agents (read /tmp file)
      → synthesis → markdown report + memory files
```

## Error Handling

- Нет JSONL файлов → "Нет истории диалогов для этого проекта"
- Файл >5МБ → только последние 500 user-сообщений
- Невалидный JSON → skip строки
- Memory директория не существует → создать
- Пустые сообщения / только IDE-контекст → skip

## Testing

- Запустить на проекте plugin (есть данные — 6 сессий, 3175 строк)
- Проверить создание memory-файлов с правильным frontmatter
- Повторный запуск — дубли не создаются, файлы обновляются
- Проверить что /tmp файл не содержит tool_result мусор
