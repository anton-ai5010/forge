## Шаг 1: Логика «обеспечь рабочую ветку» в execute

**Файлы:**
- Изменить: `forge-plugin/skills/execute/SKILL.md` — новый раздел `### 0. Обеспечь рабочую ветку под задачу` ПЕРЕД блоком «Process state» (до записи state.yml и до любого GitHub-sync), но ПОСЛЕ выбора файла плана.

**Что делаем:** slug = basename файла плана, который execute уже выбрал на шаге «Прочитай финальный план» (без даты и `.md`). НЕ сканировать `.forge/plans/` заново. Определяем базу и состояние:

```bash
BASE=$(git symbolic-ref --short refs/remotes/origin/HEAD 2>/dev/null | sed 's@^origin/@@')
[ -z "$BASE" ] && BASE=master
CUR=$(git branch --show-current)   # пусто при detached HEAD
```

Развилка (текст-инструкция в скилле, с человеческими формулировками наружу):

1. **Уже на нужной/рабочей ветке** — `CUR == feat/<slug>` ИЛИ `CUR` начинается с `feat/`|`fix/` ИЛИ мы внутри worktree (`git rev-parse --show-toplevel` ≠ корень главного репо) → **ничего не делаем, молча продолжаем**. (Закрывает worktree-сценарий и повторный запуск на своей ветке.)
2. **Ветка задачи уже существует** — `git show-ref --verify --quiet refs/heads/feat/<slug>` истинно → `git checkout feat/<slug>` (переключиться, НЕ `-b`). Закрывает повторный запуск execute.
3. **На основной ветке** — `CUR == BASE` → `git checkout -b feat/<slug>` (git сам переносит незакоммиченные правки на новую ветку, stash не нужен). **Без объявления «ветка создана»** — молча.
4. **Detached HEAD** — `CUR` пустой → не предлагать «поверх текущей»; одной фразой: «Ты сейчас не в рабочей зоне. Начинаю задачу с чистого master, ок?» → `git checkout -b feat/<slug> $BASE`.
5. **На чужой ветке** — `CUR != BASE` и не подошли пункты 1-2 → **НЕ создавать молча**. Спросить на языке смысла, без жаргона:
   > «Ты сейчас в другой незавершённой работе — "<название задачи ветки CUR, если найдём по имени>". Новая задача — её продолжение или отдельное дело? (по умолчанию — отдельное, начну с чистого листа)»
   - «отдельное» / дефолт / повтор «делай» → если дерево грязное: `git stash -u` → `git checkout $BASE` → `git checkout -b feat/<slug>` → сказать **«Твою незавершённую работу отложил в сохранность — вернёшься, всё на месте»**. Если чисто — просто переключиться и создать.
   - «продолжение» → `git checkout -b feat/<slug>` от текущей.
6. Явно прописать: forge НЕ угадывает связанность молча — пункт 5 всегда спрашивает.

**Как проверим:** поведенческий прогон (см. чекпоинт) — на master, на чужой ветке, повторный запуск, detached HEAD.

## Шаг 2: finishing — короткий путь при «мержим» + осмысленный авто-коммит + человеческое подтверждение

**Файлы:**
- Изменить: `forge-plugin/skills/finishing-a-development-branch/SKILL.md`

**Что делаем (три правки, только локальный-мерж путь, Option 2/PR не трогаем — YAGNI):**

1. **Обойти жаргон-меню.** Step 3 показывает 4 опции. Добавить перед ним: если пользователь сказал «мержим» / «всё готово» / «влей» (явное намерение слить локально) — НЕ показывать меню, сразу идти Option 1. Меню оставить только при неоднозначном запросе.

2. **Осмысленный авто-коммит** (в Option 1, перед `git checkout <base>`). НЕ `git add -A` вслепую:
   ```bash
   if [ -n "$(git status --porcelain)" ]; then
     git status --short        # показать пользователю что будет сохранено
     # закоммитить осмысленно: tracked-правки + только относящиеся к задаче новые файлы
     git add -A && git commit -m "<задача на русском>"
   fi
   ```
   В инструкции явно: показать список файлов человеку строкой «Сохраняю эти файлы: …»; текст коммита — на русском, из названия задачи; мусор (`.vscode/`, `__pycache__/`) к этому моменту уже в `.gitignore` (Шаг 3), поэтому в список не попадёт. Anton текст коммита не подтверждает — не грузим.

3. **Человеческое подтверждение после мержа** — заменить молчаливую чистку на:
   > «Готово. Правки сохранил, задачу влил в master, рабочую ветку убрал. Ты сейчас на master.»
   Без слов commit/merge/branch.

4. **Integration-заметка** — отметить в секинии, что авто-коммит срабатывает и при вызове из subagent-driven-development; там тоже показывать что коммитится.

**Как проверим:** на ветке с несохранёнными правками «мержим» → показан список файлов → коммит → влитие → ветка удалена → человеческое «Готово».

## Execution Strategy

### Последовательно (есть смысловая зависимость)
- Шаг 3 (`.gitignore`) полезно сделать ДО Шага 2-проверки, чтобы убедиться что мусор не цепляется. Порядок: Шаг 1 → Шаг 3 → Шаг 2 → проверка. Все три — мелкие правки одного типа (markdown/конфиг).

### Делегировать субагентам
- Ничего. Тяжёлого чтения/анализа нет, всё уже исследовано в фазах 2-3.

### Чекпоинты
- Один — Чекпоинт A после всех правок, перед мержем. Дробить нечего: правки мелкие и связанные, промежуточная пауза не даст что показать.
